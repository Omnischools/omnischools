# Omnischools Senior · STPSHS Integration Specification

**Status:** Research complete. Integration design proposed. Implementation deferred pending verification with WAEC on bulk-upload format for assessment scores (the one open question that shapes the export contract).

This document specifies how Omnischools Senior integrates with the **WAEC Student Transcript Portal (STPSHS)** at `https://stpshs.waecgh.org`, a government-mandated assessment submission system that every Senior High School in Ghana must use starting from the 2024/25 academic year. It is derived from the [STPSHS Operating Manual (April 2025)](https://waecgh.org/wp-content/uploads/2025/04/STUDENT-TRANSCRIPT-PORTAL-OPERATING-MANUAL_-2025.pdf) and supporting WAEC and GES publications.

---

## 1. What STPSHS is, precisely

STPSHS is the operational arm of a curriculum reform. NaCCA and GES introduced a new SHS curriculum for 2024/25 that rebalances assessment — internal continuous assessment (portfolios, project work, practicals, end-of-term exams) now formally weighted alongside the terminal WASSCE. WAEC has rebuilt WASSCE around this: only 30% of WASSCE now tests factual recall; 70% tests higher-order thinking. STPSHS is the centralised repository where every SHS records the internal portion, semester by semester, until graduation. The data feeds the WASSCE Continuous Assessment Score (CASS) component, the GPA/CGPA on the official transcript, and the qualitative-attributes section that goes with it.

**Critically, STPSHS is the system of record for the SHS academic transcript itself.** Not a copy, not a backup — *the* record. The transcript a student or institution requests is the one STPSHS generates.

It is run by WAEC, mandated by GES, and from 2025/26 access is window-gated rather than always-open. Schools must enter scores during specified windows; outside those windows, the data is read-only.

## 2. The STPSHS data model (as captured in the manual)

What STPSHS actually stores, with field-level honesty so the integration export matches exactly:

**Student bio-data (entered once in Year 1, carried through the three-year tenure):**
- Name, date of birth, sex, photograph (200×150 px, white background, jpg/png)
- Learning area and subject combination (constrained by the curriculum's min/max subjects rule)
- Assigned class
- A unique **Assessment Reference ID** auto-generated by STPSHS — *this is the canonical student key for the whole three-year cycle*
- Guardian information, address

**Per assessment cycle (semester):**
- Year of Exam (Year 1 / 2 / 3)
- Semester (1 / 2)
- Mode of Assessment (the manual references this as a filterable dimension — the actual mode enumeration is not in the public manual; needs verification with WAEC, but plausibly maps to: classroom assessment, project work, practical work, end-of-semester examination)
- Learning Area
- Subject
- Score (numeric)

The score is the only quantitative datum captured per assessment. There is no per-assignment record, no rubric, no comment field on the score itself, no portfolio attachment. STPSHS computes GPA per semester and CGPA across semesters from these scores using a credit-hours formula that the manual references but does not enumerate.

**Qualitative assessment (free text, periodic):**
- Four categories of character qualities (not enumerated in the public manual)
- Free-text descriptions per category

**Outputs:**
- Per-semester statement of results
- Three-year transcript on completion
- Head of School remarks field on the transcript

## 3. The integration constraint that shapes everything

The manual documents three data-entry paths into STPSHS:

| Data | Manual entry | Bulk upload | API |
|---|---|---|---|
| Student bio-data | ✓ (Register Student form) | ✓ (Excel template, downloadable) | ✗ |
| Assessment scores | ✓ (Capture Per Subject / Per Student) | **Not documented** | ✗ |
| Qualitative assessment | ✓ (free-text form) | ✗ | ✗ |
| **Download captured scores** | — | ✓ (admin and teachers can download) | ✗ |

The asymmetry is the whole story. **STPSHS lets data out as a downloadable file but only lets assessment-score data in via manual teacher entry.** That single fact decides the integration shape.

Three implications follow:

1. **There is no real-time sync.** Anything that calls itself a sync to STPSHS would be misleading — at best it is a CSV export the school administrator uploads on STPSHS's terms. Most realistically, until WAEC publishes a bulk-score upload path or an API, integration means *eliminating double entry by making the Omnischools UI the canonical place to enter, then exporting an STPSHS-formatted artifact when the submission window opens.*

2. **The bio-data path is the proven integration point.** Omnischools can already drive STPSHS student registration via the Excel template — that's the lowest-risk first integration. A school using both systems would register students *once* in Omnischools, then download the STPSHS-format bulk-upload sheet ready to paste.

3. **The score-submission path needs verification.** Either WAEC has an undocumented bulk-upload feature, or they don't. This is the **one open question** that shapes the contract; it is worth a direct call to WAEC's ICTD support team (0302 208201-9, ext. 2077, per the manual) before committing to a specific export format.

## 4. What STPSHS deliberately does not do

This is the section that defines the Omnischools opportunity. Going through the manual systematically and listing what is *absent*:

- **Parent or student access.** STPSHS is a school-staff portal. Parents cannot see their child's continuous assessment scores there. Students cannot either.
- **Attendance.** No attendance module. STPSHS has no view of who was actually in class.
- **Fees and finance.** Out of scope entirely.
- **Lesson planning, scheme of work, syllabus tracking.** Nothing.
- **Per-assignment granularity.** A student scored 78 in Mathematics for Semester 1 Year 2 — what *components* (which assignments, project, practical, exam) constituted that 78 is invisible to STPSHS. Only the final-mode score lives there.
- **Formative assessment workflow.** Rubrics, comments, portfolio uploads, draft-and-revise cycles, peer review — none of these. Yet the new curriculum demands portfolios and project work; the assessment of them is what produces the score that goes into STPSHS.
- **Communication.** No SMS, no WhatsApp, no parent notifications, no in-app messaging.
- **Mock examinations.** STPSHS captures internal CA scores during the three-year cycle, not mock-WASSCE results. A school's mock-exam programme is entirely outside it.
- **Analytics across schools.** STPSHS holds one school's data per account. Cross-school comparison, regional aggregation, GES district views — none of this lives in STPSHS.
- **Anything for KG / Primary / JHS.** STPSHS is SHS-only.
- **Year-round availability.** The window-gating from 2025/26 means STPSHS is read-only for much of the year.
- **Continuity of historical data before 2024/25.** STPSHS starts with the new curriculum cohort. A school's pre-existing student data must be migrated in (or simply not — older cohorts use the legacy WASSCE flow).

Every item on that list is a place Omnischools Senior provides value STPSHS does not.

## 5. Positioning: how Omnischools Senior relates to STPSHS

The positioning that follows naturally:

> **STPSHS is the regulator's ledger. Omnischools is the school's operating system.**

The analogy I'd use with a Headmaster: STPSHS is to a school's grades what GRA's online portal is to a business's PAYE — the regulator's authoritative destination. The business does not *run* on the GRA portal; it runs on its accounting system, which generates the PAYE submission. Omnischools is the accounting system; STPSHS is the GRA portal.

Concretely, three claims define the positioning:

**1. Teachers enter scores once, in Omnischools, in a UI designed for the work — not for compliance.** The Omnischools gradebook captures the assignments, project drafts, practical observations, comments, and revisions that *produce* the eventual STPSHS-mode score. When the STPSHS window opens, Omnischools generates the export sheet pre-populated with the final scores; teachers do not re-enter them.

**2. Omnischools holds the operational context STPSHS structurally cannot.** Lesson plans, attendance correlations with score dips, parent communication about a struggling student, the portfolio of project work that supports a score, mock-exam results as a predictor of WASSCE qualification. The score in STPSHS is the *outcome*; everything that produced it lives in Omnischools.

**3. Omnischools serves everyone STPSHS does not.** Parents, students, KG/Primary/JHS schools, GES Oversight. None of these have a STPSHS story; all of them have an Omnischools story.

What this is **not** is competing with STPSHS. The integration spec must make it impossible for a school to perceive Omnischools as an alternative to STPSHS — STPSHS is non-optional, and any product that suggested it could replace the government portal would be unsellable. Omnischools wraps STPSHS; it does not displace it.

## 6. Integration design

Three integration surfaces, in order of build priority:

### 6.1 Student bio-data export (low-risk, do first)

When a Year 1 student is registered in Omnischools, the system generates an STPSHS-format Excel row matching the bulk-upload template (downloadable from STPSHS). Once a class is fully registered in Omnischools, an admin downloads the STPSHS-ready Excel file and uploads it through the STPSHS bulk-upload flow. The STPSHS-assigned **Assessment Reference ID** that comes back is then stored against the Omnischools student record — this becomes the canonical link between the two systems.

Open question to verify: whether STPSHS's bulk-upload returns the Assessment Reference IDs in a downloadable file after upload, or whether the admin must read them back per-student. The former is implementable; the latter is fragile.

### 6.2 Assessment score export (the central case, needs WAEC verification)

The Omnischools gradebook is the teacher's daily workspace. It records per-assignment scores, rubric attainment, project milestones, practical observations, and end-of-semester examination marks. When the STPSHS submission window opens, Omnischools:

1. Aggregates the term's work into the STPSHS modes of assessment (the specific aggregation rules depend on which modes STPSHS recognises — needs verification).
2. Computes the final per-mode score per student per subject in the form STPSHS expects.
3. Generates either a CSV in STPSHS's expected format (if bulk-upload is permitted) **or** a per-teacher printable score sheet matching STPSHS's Capture-Per-Subject UI (the fallback if scores must be entered manually). The fallback is degraded but eliminates the *computation* burden even if it does not eliminate the data-entry burden.

This is where the verification call matters most. If WAEC has an undocumented bulk-score upload, Omnischools generates that file. If not, Omnischools at minimum gives the teacher a paper or PDF sheet they tick through screen-by-screen, which is much faster than re-deriving the scores.

### 6.3 Qualitative assessment export

Teachers enter qualitative comments in Omnischools (within a structured rubric tied to the four STPSHS character-quality categories). At submission time, Omnischools generates the STPSHS-ready text per student per category — copy-paste into the STPSHS qualitative-assessment form. Low-volume, low-frequency, an acceptable manual step.

### 6.4 What is not integrated, deliberately

- **STPSHS-generated transcripts and statements of results.** These are STPSHS's authoritative output and the school accesses them inside STPSHS. Omnischools shows the *Omnischools-side* report card (which has more in it — parent comments, attendance, project portfolios) and links out to STPSHS for the official document.
- **STPSHS qualitative-assessment free-text editing.** Once submitted, qualitative text is owned by STPSHS. Omnischools does not try to round-trip it.

## 7. Implications for the four affected Omnischools Senior surfaces

I built these surfaces against an assumption STPSHS did not exist. Reworking them is real work, but each rework is a strict improvement now that the position is clearer:

| Surface | Current state | Required rework |
|---|---|---|
| `schoolup-wassce-subject-teacher.html` | Teacher enters WASSCE-style scores directly | Reframe as the daily gradebook — per-assignment, rubric-aware. Add an "Export to STPSHS" affordance at term end. The teacher's view does not center on WASSCE scoring; it centers on the work of teaching, with STPSHS submission as one downstream action. |
| `schoolup-wassce-student-readiness.html` | Predicts WASSCE qualification from internal scores | Now meaningfully different from STPSHS — STPSHS holds the CA scores but does not project WASSCE outcomes. Position as the **predictor STPSHS structurally cannot do**, using internal + mock + STPSHS-submitted scores. |
| Student detail (performance section in school detail and student profile) | Shows internal continuous-assessment scores | Add an "STPSHS submission status" panel — what's been submitted, what's pending the next window, the Assessment Reference ID, and a link to view the official transcript in STPSHS. |
| `schoolup-oversight-academic-performance.html` (Oversight) | Internal/continuous performance section sources from school gradebooks | Source for Oversight: Omnischools gradebook (live) **for non-STPSHS data**, plus STPSHS-submitted scores via the same export the school generates (since a GES district director cannot read STPSHS directly per-school, but Omnischools knows what was submitted). The coverage caveat carries through cleanly — internal performance Oversight is honest about the share of schools using both systems. |

None of these are rebuilds. Each is a reframe of an existing surface with a new bottom-right "STPSHS" panel or status strip and a clearer story above it.

## 8. The open question that gates implementation

Before any of section 6 or 7 ships, one fact needs confirming directly with WAEC:

**Does STPSHS accept bulk upload of assessment scores in any form?** Either an undocumented Excel-template path analogous to bio-data registration, or a per-teacher CSV upload route accessible through the Capture-Per-Subject UI.

If yes — Omnischools generates that file, and the export is genuinely a one-click submission.
If no — Omnischools generates a printable per-teacher score sheet matching STPSHS's UI flow, which is a meaningful improvement on memory-and-typing but is not zero-friction submission.

The right next action is a call to **WAEC ICTD Support, 0302 208201-9 ext. 2077** (the contact listed in the manual), framed as a vendor enquiry: "We are building school-management software and would like to confirm what bulk-upload paths exist for assessment scores beyond the documented student-registration template." A 15-minute conversation resolves the contract. Until then, the surface rework above is on hold and the spec stays at "design proposed, awaiting confirmation."
