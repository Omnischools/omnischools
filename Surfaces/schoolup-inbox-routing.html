When the inbox has owners

The MVP1 inbox treated parent threads as a single shared list — whoever opened it first replied. That works at small scale; at a school with multiple staff handling parent communications, threads need explicit ownership. Three states: the inbox view with assignment indicators visible, the reassign drawer for handing off a specific conversation, and the routing rules settings page where admin defines which threads go to whom by default.

Why this layout

  • 5-tab inbox header replaces the MVP1 single-list view. "Mine" (italic gold-emphasised) / Team / Unassigned (warn-tinted with attention) / @Mentioned me / Closed. Joyce defaults to "Mine" because that's her work; the other tabs are visible but secondary.
  • "Mine" tab uses italic gold "Mine" wording in the active pill. Same Fraunces italic + gold convention that signifies user-relevant values throughout the system. The italic communicates "this is yours specifically" — different from a flat "Mine" label.
  • Unassigned tab in warn state with bordered count draws attention without being a notification badge. Unassigned threads are an operational risk — they belong to no one and could fall through the cracks. The warn state nudges admin to triage them.
  • Lede line summarises the inbox state precisely: "7 assigned to you · 12 across the team · 3 unassigned · sorted by latest activity." Three numbers in one sentence; admin reads this and knows where things stand without scanning the tabs.
  • "Routing rules →" CTA in the page header opens the rules settings (state C). Routing rules are accessible from the inbox because that's where admin notices a rule needs adjusting ("threads keep going to Mary who's on leave"); putting them in deep settings would create distance.
  • Thread rows show assignee chip below the preview text. Three states: Mine (green-tinted with mini avatar), Colleague (neutral with their initials), Unassigned (terra-tinted with "?" avatar). Joyce can scan the list and see at a glance which threads are hers, which belong to teammates, which need ownership.
  • Sarah Amponsah's "URGENT" thread is unassigned — a parent reporting a playground injury. The pairing is operationally meaningful: high-stakes topic + no owner = highest triage priority. The terra urgent tag plus terra unassigned chip stack visually.
  • Topic tags colour-coded: billing gold-bg, attendance/schedule neutral, academic green-bg, urgent saturated terra. Topic detection drives routing (state C) so the same colour vocabulary appears at every level — message preview, rule destination pill, rule condition pill.
  • Each thread row has the WhatsApp green W badge over the bottom-right corner of the avatar. Same convention as MVP1 inbox; signals the channel without taking dedicated column space.
  • Section dividers between "Active · today" and "Earlier this week" with right-aligned counts. Time-based grouping rather than just chronological scroll — admin's mental model is "what's hot today vs what's older."
  • Conversation header shows parent + child + WhatsApp number. Same as MVP1 inbox. The new addition is the assignment metadata bar below the header in gold-tinted background.
  • Assignment bar carries provenance: "Joyce Owusu-Ansah · you" + "Auto-routed by rule 'Billing topics → Joyce' · 2h 14m ago". Joyce sees why she got this thread, not just that she has it. If the rule is wrong (this isn't really a billing topic), she knows where to push back.
  • "Reassign →" CTA in the assignment bar opens the reassign drawer (state B). The button has a gold border on white background — distinctive but not loud. Reassignment is common; the affordance has to be visible without dominating.
  • Reassignment events appear inline in the message stream. "Auto-routed to Joyce Owusu-Ansah · billing topic detected · 9:43 AM" rendered as a centred capsule between messages. System events are part of the conversation history, not hidden in a separate audit log. If an auditor asks "when did Joyce take over?", the answer is right there in context.
  • Internal note in gold-tinted dashed-bordered card. Florence left a note for Joyce: "Patricia is reliable — last term she also paid in 2 instalments, both on time." Notes are staff-only knowledge attached to the conversation. Same dashed gold visual as other "staff-private" content elsewhere in the system.
  • Internal note has author + timestamp + "in this thread" attribution. "@Florence · 10:03 AM · in this thread" — tells Joyce who wrote it, when, and confirms it's scoped to this conversation. The @ prefix matches the @-mention convention from the composer.
  • Composer has two tabs: "Reply to Patricia" and "Internal note". Active tab in surface-white; note tab when active flips to gold-tinted background. The composer changes shape based on what you're writing — replying to parent vs writing a staff-only note. Visual switch prevents the costly mistake of accidentally sending an internal note to the parent.
  • "@" mention hint in the composer footer. "Type @ to mention a colleague · they'll be notified but the parent won't see the mention." Mentions pull colleagues in without transferring ownership — different from reassignment. The hint normalises the @ syntax from messaging apps.
  • What's not in this state: No SLA timers ("respond within 4 hours"), no priority levels (urgency comes from topic tag, not a separate field), no "snooze thread" ability (deferred messages are inbox-zero anti-pattern), no thread-to-thread merging. Keeping the inbox simple while adding ownership.

Why this layout

  • Drawer slides in from the right with dimmed background. Same drawer pattern as edit-request approval, follow-up, send-reminders. Visual continuity across all assignment-style transactions in the system.
  • Drawer head: "Hand off to a colleague" headline with parent + student + class context in the lede. The lede also tells Joyce what will happen: "the new assignee will be notified and Patricia will see the next reply from them." Honest preview of consequence before action.
  • From → to preview at the top of the drawer body. Two avatar+name+role cards with a gold arrow between them. Joyce sees the transaction visually before scrolling to choose. The "you" tag on her own card disambiguates which side of the transfer she's on.
  • The "to" card updates as Joyce picks a colleague. Currently shows Florence Addo because she's selected (suggested). If Joyce clicks Daniel Owusu, the "to" card flips to him. Live preview of the choice at the top of the drawer; doesn't require scrolling back up to confirm.
  • Colleague list with availability indicators. Online (green dot) / Away (warn dot · "23 min") / Offline (grey dot). Helps Joyce avoid handing off to someone who can't pick it up — passing a thread to an offline colleague means the parent waits longer.
  • "Suggested" reason tag in green saturated pill on Florence's row. The system suggests her because the conversation has shifted to fee exemption (a headmistress decision). Suggestion based on what the message is now about, not just role. Lighter-touch routing than rules; the suggestion can be ignored.
  • Each colleague's role line shows specialty: "Headmistress · fee waivers & exemptions" / "Asst. headteacher · JHS 2A homeroom" / "Science teacher · JHS 1B homeroom" / "Asst. headteacher · math" / "Social studies · JHS 3A". Bold-emphasised specialty tells Joyce why each colleague might or might not be a good fit at a glance.
  • Selected row has gold-tinted bg + outlined avatar. Visual signal of "this is who you're transferring to." Combined with the from→to preview at the top, Joyce has two reinforcing signals of her choice.
  • Handoff note section optional but pre-filled with realistic context. "Patricia is asking about a partial payment arrangement... she's now asking about a fee exemption for next term — that's a headmistress decision, not mine. Florence, can you take it from here?" The pre-fill demonstrates the kind of note that makes handoffs work; not just "transferring to you" but "here's the situation and what they need."
  • "Notify Florence by WhatsApp" toggle defaults on. Florence will get a brief WhatsApp from Omnischools's number alerting her to the new assignment + the handoff note. Real-time notification means Florence can respond from anywhere, not just when she's at her desk. Particularly useful in Ghanaian schools where headteachers spend much of their day around campus, not at a computer.
  • "Add reassignment to thread audit log" toggle is forced-on (greyed) with explanatory text "always on · cannot be disabled · part of the conversation record." Audit cannot be disabled because thread ownership history matters when complaints or disputes arise. Honest about the lock.
  • Two action buttons in footer: "Reassign without note" + "Reassign with note →". The note is genuinely optional; admin can transfer without one if context is obvious. The gold-primary CTA is the with-note version because that's the recommended path. Two buttons, two paths, neither hidden behind a menu.
  • What's not in the drawer: No "transfer ownership for X days" temporary assignment, no "request acceptance" workflow (would be too formal for school context), no "transfer to multiple people" (one owner per thread is the constraint), no "transfer history" displayed inside the drawer (it's in the conversation stream and audit log).

Why this layout

  • Crumb + "Routing rules" + lede explaining the evaluation model: "evaluated top to bottom · the first matching rule wins · the fallback catches everything else." The model is stated up front; admin reads two sentences and understands how rules work.
  • 4-tile activity stat strip at the top. "184 threads routed in 7 days · 96% by rule · 4% manually" / Most-used rule (Billing → Joyce) / Reassignments after auto-route (8/184 = 4.3%) / Hit fallback (7 threads = 3.8%). Stats answer "are my rules working?" Admin sees the system's behaviour at a glance.
  • "Rules are 95.7% accurate" derived from the reassignments-after-route count. If 8 threads got reassigned after auto-routing, those 8 went to the wrong person initially. Honest accuracy metric the admin can act on — if the number climbs, rules need tuning.
  • "3.8% within healthy range" on the fallback stat. A small percentage of unmatched threads is expected; if the fallback rate climbed to 30%, rules would need expanding. Setting expectations for what a healthy rule system looks like.
  • Rules listed top-to-bottom with order numbers in gold-tinted tiles. Order matters — the first rule that matches wins. Drag handles on the right let admin reorder. Ordering is part of the rule semantics, not just visual.
  • Rule statement in natural English with inline pills. "When the message is about [Billing or payments] route to [Joyce Owusu-Ansah]." The "When ... route to ..." sentence reads naturally; the pills are the user-tunable parameters. Reads like a sentence, edits like a form.
  • Pill colour-coding: condition pills gold-bg, destination pills green-bg. Same vocabulary as everywhere else — gold for "this is the trigger" / green for "this is the outcome." Visual consistency across the system.
  • Destination pills include avatar + name, not just the staff member's name. The mini avatar in green saturated circle echoes the assignee chips elsewhere. Visual identity for the colleague tied to their role across the system.
  • Compound conditions: rule 2 has two condition pills ("JHS 2A" + "Attendance or academic"). The natural-English connectors ("and topic is", "and") make compound rules readable without using AND/OR boolean syntax. Linguistic, not algebraic.
  • Rule 4 has a non-obvious destination structure: route to Florence + "notify all admins". The "+ notify all admins" is a secondary effect of the rule beyond simple assignment. Urgent/safety rules need extra distribution because no one person should bear sole responsibility for a child injury thread.
  • Each rule shows three meta-data points: "Triggered N times · 7d" / "Last fired X ago · who" / "Reassigned after route: M/N · pct." Per-rule effectiveness data. If rule 1 has 62 firings and only 2 reassignments, it's working well. If a rule had 50 firings and 30 reassignments, something's wrong with the rule's logic.
  • Per-rule toggle on the right lets admin disable a rule without deleting it. Useful for "Mr. Owusu is on leave for 2 weeks · disable his rule until he's back" without losing the rule definition. Disable ≠ delete; reversible state for temporary changes.
  • Fallback rule rendered in dashed-border, bg-tinted card with "∞" order number (no real position). Visual signal that the fallback is special — always last, always runs, can't be deleted. The dashed border distinguishes it from active reorderable rules.
  • Fallback configured as "Leave unassigned + notify Florence Addo". Smart default — unmatched threads stay visible to admin (in the Unassigned tab) AND admin gets a heads-up to triage them. Both visibility (in inbox) and urgency signal (notification) at the fallback level.
  • "+ New rule" CTA in the page header opens a rule creation flow (out of scope for this design). Rules are admin-creatable — schools can adapt routing to their org structure. The 4 rules shown are reasonable defaults the system might suggest at onboarding.
  • "Test a message ↗" CTA lets admin paste a hypothetical parent message and see which rule would match. Debugging tool; useful when a rule isn't behaving as expected. The ↗ suggests it opens a side surface (probably a small dialog with input + matched-rule preview).
  • What's not on this page: No A/B testing of rules, no machine learning rule suggestions, no automatic rule generation from past assignment patterns. Manual rules with visible behaviour; the admin authors and adjusts.
  • What's also not here: No time-based rules ("route to X during business hours, Y after"), no parent-language rules ("Twi speakers go to..."). Both could be future enhancements. The 4 default rules cover the most common Ghanaian school scenarios.
