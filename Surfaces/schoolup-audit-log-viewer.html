The audit log viewer. Every change to school data has been recorded since onboarding day 1 — admissions decisions, void/refund actions, fee adjustments, attendance edits, settings changes. This is the surface that collects all of those into one queryable place. The school can answer "who did this and when?" without needing the developer's help. When the Omnischools account is shared between two admins, the audit log is what makes that arrangement workable — anything one of them does is visible to the other.

Why this layout

  • Lives under Settings → Audit log as a sub-nav item · sister to School profile, Users & roles, Billing rules, SMS & WhatsApp, Data & retention, Security. Audit log is settings-adjacent: the school's record of itself.
  • Hero pillar leads with editorial summary: "214 events across 3 actors in 11 modules." Then meta: "Most active: Joyce 138 (mostly billing) · Florence 67 · system 9 · 0 anomalies flagged." One sentence answers "is anything weird going on today?".
  • 4-tile strip beneath the hero: Term-to-date count (1,847) / Sensitive ops (23, warn-tinted) / System integrity (✓ green, "log sealed nightly"). "Sealed nightly" is a critical phrase — the audit log is hashed at midnight so any subsequent tampering would be detectable; bank-grade integrity guarantee.
  • Filter card with 4 dimensions: Module (9 pills with counts) / Actor (4: Anyone, Florence, Joyce, System) / Severity (4: Any, Routine, Sensitive, Anomaly) / When (4: Today active, 7 days, This term, Custom). Severity is a first-class filter — admin can isolate just the sensitive ops in one click; this is how they answer "any voids today?" without scrolling.
  • Free-text search box with example placeholder: "void", "Patricia Boateng", "GHS 2,000". Search across all entry text; currency strings are searchable so admin can find a specific transaction by amount when they don't remember the date.
  • Each entry is a card with 3-column grid: timestamp + ago / body / action. Timestamps in mono font, "ago" relative time below. Mono font for timestamps and IDs — visual rhythm of the column makes scanning easy.
  • Entry headlines are subject-verb-object sentences in plain English: "Joyce Owusu-Ansah reconciled 47 MoMo payments in one bulk action · totalling GHS 14,580." Reads like the school's own diary; parents' names and amounts are bold so they jump off the page.
  • Severity-tinted backgrounds: routine is plain white, sensitive is warn-tinted (subtle yellow), anomaly is terra-tinted (subtle red). Visual triage at the row level; admin glancing at the feed sees the warn rows first.
  • Bulk events get a gold left-border + "⚡ BULK · 47" pill · this is the operationally interesting category. Joyce's 47-payment reconciliation reads as one event with 47 inside it, not 47 separate entries that would drown the log. Same for the 41-student attendance mark.
  • Bulk events have a gold-tinted summary block: "Matched 47 of 49 MoMo entries · 2 unmatched (saved as suspense)..." with a "View all 47 receipts →" expand link. The detail is one click away; the bulk parent entry stays scannable.
  • Sensitive events show inline diff blocks: "Status: paid → voided." Mono diff format like git; admin sees what changed without opening the entity.
  • System events have actor "System" not a person — Daily SMS fires, scheduled sends, etc. System is a first-class actor; admin can filter just system events to audit automation.
  • Anomaly entry in terra: "⚠ login attempt outside normal hours · 4 failed attempts in 6 minutes · IP unfamiliar" with terra "Investigate →" action. Anomalies are the alert mechanism — admin doesn't need to monitor for security issues, the system surfaces them and demands attention.
  • IP and device metadata on every entry: "IP 197.250.10.182 · desktop · Chrome 119" (or "mobile · Android"). Forensic context; if Joyce claims she didn't make a change, admin can see whether it came from her usual IP and device.
  • Day markers chunk the feed: "Today · Wed 22 Oct · 214 events" then "Yesterday · Tue 21 Oct · 187 events." Counts at the boundary show pacing across days; sudden spikes become visible.
  • Per-row action buttons are contextual: bulk reconciliation gets gold "Drill in →", void gets neutral "View receipt", attendance gets "View register", anomaly gets terra "Investigate" with white text. The action follows the entry semantics; admin doesn't have to decide what to do, the system suggests.
  • Auto-refresh every 30s noted in the region head. The feed is live; if Joyce is doing something right now, Florence sees it within half a minute. This matters for shared-account scenarios where two admins are both working.
  • "append-only · entries cannot be edited or deleted, even by school owners" stated explicitly in the lede. The audit log's value collapses if it's editable; this is the single most important property and it's stated where every viewer will see it.
  • PII tombstoning at retention horizon · audit rows are never deleted but PII fields (subject name, phone, NHIS, address) are pseudonymised to [REDACTED-YYYY-MM] markers when the underlying record is purged under module retention policy (Academic 7yr · Financial 5yr · Operational 3yr · all configurable above regulatory floors). The who did what when survives forever; who was affected ages out with the underlying record. The 22 Sep 2022 example row above shows what a tombstoned entry looks like in the viewer · hover the marker to see the redaction reason. Each redaction is itself an append-only entry in the audit_pii_tombstones table — the redaction itself is auditable.

What's preserved per event

  • Drill panel opens to a 4-section breakdown: Actor / Action / Affected records / Integrity fingerprint. The four classical questions of forensic accounting: who, what, what was changed, can we trust the record? Same shape for every event, so admin learns the structure once.
  • Actor card includes 2FA verification state: "2FA verified ✓" · because Joyce passed her second factor at login, the action is attributed to her with high confidence. Without 2FA, the audit log's attribution is weaker; this is one of several reasons 2FA matters (file 4 of this batch).
  • Action section uses key-value grid: Module / Verb (in caps · machine-readable) / Severity / Trigger / Duration / Reversibility. Reversibility is named explicitly: "Reversible · individual payments can be voided · void itself produces a new audit event." Reversal is auditable too; the log never shrinks, only grows.
  • Verb in mono caps: BULK_RECONCILE. Machine-readable verb space exists because audit data needs to be exportable for external analysis (file 3 of this batch handles export). The display layer renders these in plain English in the feed; the underlying data uses a controlled vocabulary.
  • Affected records list is scrollable, max 160px tall, with one suspense row in warn: SUSP-2024-008 · "Unmatched · No invoice matched ref EFE49P12X" with terra warning state. The 2 unmatched entries from the bulk are preserved as part of this event; admin knows exactly which 2 entries didn't process and can investigate from here.
  • Each affected-record row shows ID, name+context, amount, final state. The row format is consistent with billing tables elsewhere in the app — same pattern, same widths. Visual continuity across modules; admin doesn't relearn the table.
  • Integrity fingerprint section shows hash chain: Event ID · Recorded at (ISO timestamp with timezone) · Event hash (sha256 fingerprint) · Prev event (the audit log is a hash-chained list, like a blockchain) · Day seal · Day hash. This is how the system proves "this entry hasn't been altered": any change to past data would break the hash chain and be detectable by re-computing.
  • "Day seal: Pending · seals at 23:59:59 GMT today" · means today's events are still being added to the chain · at midnight the day hash is computed and posted (anti-tamper checkpoint). Prevents same-day tampering: even an admin with database access couldn't go back and edit today's events without breaking tomorrow's seal.
  • Actions on the drill panel header: "Open in billing →" (jump to the actual billing module to see the result of the action) and "Export this event" (download a JSON or PDF with everything from this drill panel). Export per-event matters because banks and lawyers sometimes want a single event's full record without the rest of the log.
  • "This event has been sealed with the day's audit hash and cannot be modified" stated in the lede. The seal IS the value; without it the audit log is just a journal, not legal evidence.
