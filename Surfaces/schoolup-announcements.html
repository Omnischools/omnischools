Omnischools · Announcements

Saying it once, reaching everyone

The composer for school-to-parent announcements. SMS is the workhorse for MVP1 — universal reach, feature-phone-friendly, predictable cost. WhatsApp ships in MVP2 as the second pillar: cheaper per conversation, richer media, but requires opted-in recipients and Meta-approved templates. School-name suffixes ("School", "JHS") are stripped from SMS sign-offs to conserve every character.

Why this layout

  • Four-step structure (Audience → Channels → Message → Timing) makes the composition order explicit. Audience first because everything else depends on it — recipient count drives cost, channel availability, and message constraints.
  • Audience as chips with grouping dividers. "All school" sits separate from "JHS 1, 2, 3" which sits separate from "All teachers" — three governance categories. Multi-select is allowed (JHS 1 + JHS 2 selected); the chip count updates the recipient total live.
  • Audience summary block in gold reads back the selection in plain language: "Sending to JHS 1 and JHS 2 guardians — 4 sections combined. Excludes JHS 3 and teachers." Plus the recipient count (118) in a Fraunces accent. Admin sees what they've selected and what they've excluded — both matter.
  • Channel tiles with cost shown per channel. SMS shows "~GHS 0.05 per send"; in-app shows "Free · 41% have app" — the trade-off is explicit. Admin chooses based on reach (SMS gets to everyone; app only reaches the 41% who've activated).
  • Email tile is disabled with "Coming MVP2" label. Honest about what's not shipping yet, while making space for the future. The disabled state matches the "MVP2" coming-soon pattern from the reports module.
  • Subject + body as separate fields. Subject shows in-app (push notification heading) and email; body is the SMS content. Different channels, same composer — the data model handles the multiplexing.
  • Character meter is the load-bearing detail. "136 / 160 chars · 1 segment · sign-off adds 17 chars" — the meter accounts for the auto-appended sign-off so admin doesn't write a 155-char body that becomes a 174-char send (= 2 SMS segments = double cost). The meter shows 153 as the actual total.
  • School-name suffix stripped for SMS sign-off. "Christ the King JHS" displays everywhere — receipts, in-app, dashboards — but the SMS sign-off uses "Christ the King" only. "JHS", "SHS", "School", "Basic School" all strip; "Academy", "College", "Institute", and religious designations preserve. The savings are real: 4-7 chars per SMS, which is the difference between staying under 160 and running into a second segment. Admin sees what gets stripped via the small label below the sign-off so it's transparent, not magic.
  • Four-channel picker: SMS (MVP1), WhatsApp (MVP2), In-app push (MVP1), Email (MVP2). Two-by-two grid keeps each tile prominent. SMS is the default selection because it has universal reach and doesn't require parent opt-in. WhatsApp shows as a real future option with realistic cost figures and opt-in counts so admin can plan ahead.
  • WhatsApp tile shows opt-in count — "84 opted in" of 118 selected recipients. WhatsApp Business API requires recipients to have opted in; the tile makes that constraint visible. In MVP2, admins will use SMS for non-opted-in recipients and WhatsApp for opted-in, automatically.
  • WhatsApp cost ("~GHS 0.18 per conversation") uses Meta's actual conversation-based pricing, not per-message. A "conversation" is 24 hours of back-and-forth; one outbound template plus any subsequent two-way messaging counts as one conversation. The cost per recipient looks higher than SMS but each conversation can carry multiple messages, attachments, and quick-reply buttons — net cheaper for richer interactions.
  • Schedule is "send now" by default. Most announcements are immediate. Schedule reveals a date/time picker (not drawn — separate flow). Two clean options, no decision paralysis.
  • Emergency toggle in terracotta with explicit consequence text. "Bypasses quiet hours, uses priority delivery. Use only for genuine emergencies." The toggle is off by default and the language discourages casual use — emergency status has system consequences (priority queue, possibly higher cost from SMS provider) and shouldn't be casual.
  • Cost summary in navy prominently in the right rail. Per-channel breakdown plus total. The total is the headline; admin sees this before clicking Review & send.
  • SMS preview as a phone bubble showing exactly how the message will arrive on the parent's phone. From "Omnischools" (the SMS sender ID), the body, the auto-appended sign-off. Preview = ground truth — what's shown is what sends.
  • Template picker on the right lists common school announcements. The "PTA meeting reminder" tile has a gold left-border because it was just used as the starting point. Click any other template to swap in its content.
  • Sticky footer with the recipient count and cost echoed. "Ready to send to 118 recipients · estimated cost GHS 5.90". Even if admin scrolls back up to edit the body, the bottom bar reminds them of the consequences. Two actions: Save as draft (no commitment) and Review & send (proceeds to confirmation drawer).
  • Save as draft is everywhere — page header and footer. Composing under pressure isn't always linear; admin should be able to step away and come back.

Why this layout

  • Final confirmation, not "are you sure?" The drawer's job is to show admin exactly what's about to happen — message text, recipients per group, cost per group, timing. The decision is enabled by full information, not by abstract warning.
  • Message preview at the top in the same SMS phone bubble shape as the composer. Identical visual format means identical interpretation — what admin saw composing is what admin sees confirming.
  • Recipients broken down by group with per-group cost. "Guardians of JHS 1A — 28 sends — GHS 1.40" lets admin verify each group is correct. If one of the groups looks off (wrong class, unexpected count), admin sees it before sending.
  • Total row with a heavy navy underline — same pattern as receipt totals. The total is the consequence; everything above is the breakdown.
  • Deduplication note explains the count. A guardian with two children gets one SMS, not two. Without this note the math wouldn't add up to admin (118 students but only 96 sends?). The note prevents support questions.
  • Timing block in green tells admin when delivery happens. Not "sent" but "starting to arrive within 30 seconds" — sets accurate expectation. SMS gateways take 4-6 minutes for full delivery to a 100+ recipient blast; admin shouldn't expect instant.
  • Confirm checkbox is required. Unchecked, the send button is disabled. The "I understand this cannot be undone" framing is the single piece of friction that prevents accidental sends.
  • Cost deduction is mentioned in the confirm block — "the cost (GHS 5.90) will be deducted from the school's SMS balance." Surfaces the financial reality. Schools running low on SMS balance see this and know to top up.
  • Edit message button goes back to the composer with everything preserved. Confirmation drawer doesn't lose state — admin can iterate.
  • Send button anchored to specific count: "Send to 118 recipients →". Not "Send" alone — anchored to the number so admin knows exactly what they're authorising.

Why this layout

  • SMS balance prominent in the page header — "SMS balance · GHS 432.00." Admin sees the running balance every time they visit the page. Encourages topping up before it hits zero rather than discovering empty mid-emergency.
  • Filter pills with counts: All / Sent / Scheduled / Drafts. Same shape as the audit log filters. Most common scan is "what's gone out lately" (default All view, sorted by recency).
  • Scheduled rows tinted gold with "Scheduled" tag in the When column. Sit at the top because they're the most consequential — admin needs to verify or cancel before the send time arrives.
  • Emergency rows tinted soft red with "Emergency" tag. The flood-closure announcement reads as significant in the timeline; auditor scrolling through the year sees emergency events stand out.
  • Delivery percentage with progress meter. 99% / 96% / 100% — most messages don't reach 100% because of inactive numbers, network failures, or full inboxes. The 88% sports day row shows that lower delivery rates are flagged in gold and prompt a "Resend" action.
  • "Resend" button only appears on rows below a delivery threshold. The action triggers a re-send to undelivered numbers only — not duplicating to delivered ones. Cost is reduced because only the failed attempts get re-billed.
  • Subject + preview pattern. Subject in Fraunces, first sentence of body in italic Manrope grey below. Admin scans subjects; the preview offers context without expanding rows.
  • Audience tag is a pill showing the broadcast scope and recipient count. "JHS 1 + 2 · 118" / "Outstanding · 142" / "All school · 600" — terse but informative.
  • Channel chips in mono caps ("SMS", "App") show how each announcement was delivered. A row with both means admin sent to multiple channels; admin can verify the choice was right.
  • "View" vs "Edit" actions per row. Sent rows offer View (read-only detail with delivery breakdown by recipient). Scheduled and draft rows offer Edit. Emergency rows offer View; emergencies aren't editable post-send because they may have triggered priority dispatch.
  • Default sort is recency descending, with scheduled pinned to the top regardless of date. The eye reads the upcoming first, then the past in reverse chronological — same temporal logic as a calendar.

Recommended channel strategy

How to mix SMS and WhatsApp

SMS
Use SMS for
  • ·Emergencies — universal reach including feature phones
  • ·First-time outreach to parents who haven't opted in
  • ·Authentication codes — OTPs always go via SMS
  • ·Critical balance reminders — overdue 30+ days
WA
Use WhatsApp for
  • ·Receipt PDFs — attach the file directly to the message
  • ·Achievements with photos — sports day, performances
  • ·Routine reminders with quick-reply buttons (Confirm / Decline)
  • ·Term reports — link to PDF + brief summary
Omnischools can choose automatically. When this is on, Omnischools picks the best channel per recipient — WhatsApp if they've opted in, SMS otherwise. Admin doesn't think about it; the system minimises cost while maximising reach.

Why this layout

  • Connection status banner at the top in green tells admin the integration is live. Includes the Meta verification date — schools that need to demonstrate compliance to parents or board can point to this.
  • Four-tile stats strip with the data that matters: opted-in parents, approved templates, sent this month, cost this month. The cost tile flags the savings vs SMS — 62% cheaper for the same volume — which is the business case in one number.
  • Opt-in flow as 3 numbered steps. Tap in app → Omnischools sends invite → parent replies YES. The flow is genuinely simple but worth showing because most schools and most admins haven't done WhatsApp Business onboarding before. The "Why opt-in matters" callout explains the policy reason without lecturing.
  • Templates list with status pills (Approved / Pending). Pending templates can't be sent yet — Meta typically takes 24-72 hours. Showing pending status sets expectation. Each row shows the actual template body with placeholder variables ({parent_name}, {amount}) so admin knows what gets sent.
  • "Submit new" template button opens a separate flow (not drawn) where admin writes a template and submits to Meta for approval. Omnischools manages the API submission; admin just sees status.
  • Recommended channel strategy block as the educational anchor. Two columns: "Use SMS for" / "Use WhatsApp for" — with concrete examples. OTPs always SMS is the load-bearing rule (WhatsApp's 24-hour conversation window doesn't fit short-lived OTPs, and SMS is the universal authentication channel).
  • "Omnischools can choose automatically" as an explicit feature in gold. Once both channels are set up, the system can route per-recipient. This is the real product win — schools don't think about channels, they think about messages, and Omnischools picks. Reduces cost without admin friction.
  • Receipt PDFs via WhatsApp is genuinely transformative. Currently SMS sends a link the parent has to tap, then a browser opens, then PDF loads. WhatsApp attaches the PDF directly — three taps becomes one. Worth highlighting as a use case because it's tangible.
  • This whole screen is MVP2 scope. Designed now to validate the data model (opt-in tracking, template registry, conversation costs, channel routing) and to show where things land. Ships when WhatsApp Business API integration is built — probably 2-3 weeks of focused work after MVP1.
