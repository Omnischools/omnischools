Mr Mensah is not one thing

In Ghanaian SHS, a teacher is rarely just a teacher. Mr Mensah teaches Mathematics, leads the Maths department, runs Aryee House, facilitates weekly PLC sessions, and sits on the school's discipline committee. His GES rank (Principal Superintendent) is separate from any of those roles — it's a national civil-service ladder he climbs over decades. The staff record must hold all of this without becoming a Christmas tree. Density without overwhelm is the design challenge.

Why this matters in Omnischools: Mr Mensah's GES rank determines his salary scale (SSSS Level 19.1), promotion eligibility, and signing authority for official documents (transcripts, GES returns). His school roles determine what he does day-to-day. They're different facts about the same person, kept in different fields.
The principle · facts that don't collapse

The wrong design would be a single "title" field that says "Assistant Headmaster · Mathematics · Aryee House · PLC" mashed together. The right design separates rank (one-per-person, civil-service ladder) from roles (many-per-person, school-defined, time-bounded). Each role has its own scope (which classes, which House, what period). When Mr Mensah gets promoted to Headmaster in 2030, his rank changes to Deputy Director and his role changes to Headmaster — independently. The model holds.

Calls worth flagging

  • The profile header shows three identity facts at once: name, rank, primary role. The summary line beneath the name reconciles them — "Assistant Headmaster Academic · 4 active roles · teaches across 3 Forms" — so admin doesn't have to read the whole page to know the basics.
  • The rank card in the top-right is gold-tinted because GES rank is a protected fact — admins can't change it (only GES promotion processes can). Surfacing it as a read-only card communicates "this is a fact, not a configuration."
  • The 4 active roles are visually differentiated by icon colour: navy (leadership), gold-bg (teaching), green-bg (House), surface (coordination). Pattern recognition before reading.
  • "Since Sept 2014 · original role · 11 years" on the Subject Teacher row — admin sees instantly that teaching is Mr Mensah's primary identity, the leadership and house roles came later. This is contextually important.
  • The GES rank ladder visualisation shows all 10 ranks with Mr Mensah's current position highlighted, past ranks dimmed, future ranks shown ahead. Admin doesn't need this every day — but when reviewing a teacher for a school-level promotion or doing GES return paperwork, it's invaluable.
  • The CPD progress ("14 / 20 for 2025/26 cycle · 70% complete") is surfaced in qualifications because it affects licence renewal — a teacher below threshold can't renew, and admin needs to know that 6 months before the expiry.

The picker is opinionated, not free-form

4 role categories cover everything a teacher might do in SHS — picking one narrows the next field meaningfully. Within "Coordinator," the dropdown lists the common roles Omnischools knows about (PLC, Exams, Guidance, Sports, Library), with a "type a custom name" affordance for the rest. The picker is opinionated because most schools' role structures are similar enough that defaulting is faster than starting blank — but flexible enough to handle the exceptions.

Calls worth flagging

  • The "Compensation impact" field is gold-tinted as a heads-up — admin sees that adding this role triggers a payroll change. Surfacing the financial consequence prevents surprises later.
  • For roles Omnischools doesn't know carry an allowance (e.g. custom school-defined coordinator), the compensation field is empty with a "no automatic allowance — adjust on payroll if needed" helper. Don't fabricate facts.
  • The Coordinator category is the catch-all — schools have invented endless coordinator roles over the years (Cultural, Health & Safety, ICT, Library, etc.). Letting admin add custom roles within the category is the right balance between structure and flexibility.
  • "5th active role" in the footer reminds admin of role concentration — some teachers carry too many hats, and a discreet count nudges admin to consider whether to reassign instead.
  • Role start dates are required, end dates optional — most roles are open-ended ("I'm Asst Head until I'm not"). The end date gets set when the role ends, creating a clean historical record. Forward-only, append-only.

The migration · one-time backfill

For each existing staff row with a non-null role field, Omnischools inserts one row into staff_role_assignments with the same value as scope, type as "Teaching", and start_date as the staff member's join date. The original role column stays for two release cycles as read-only fallback, then dropped.

Net impact on basic-school admins: nothing visible changes immediately. Their staff records still show one role per teacher — because most basic-school teachers only have one role. The structural capacity is there, ready for the day a basic-school admin wants to add a 2nd role (e.g. "Sports Coordinator") without changing schemas. Existing data preserved, new capability available.
For the build phase

This refactor is small in code (one migration, one new table, one updated query layer) but large in consequence — it unlocks SHS, it unlocks multi-role basic-school teachers, it unlocks honest historical records. Doing it in Sprint 1 of MVP1 (rather than later) means we never have to migrate live customer data. The cost is one extra day in Sprint 1; the alternative is a painful re-architecture mid-launch.

Calls worth flagging

  • The ges_rank_id field on the staff table is nullable because basic-school staff records often won't have a GES rank set during MVP1 — admin doesn't need to fill it in unless they want to. SHS staff records require it (enforced at the UI level, not the schema).
  • The role column staying for two release cycles is a defensive measure — if the new query layer has a bug, admins can fall back to the old field. After two releases without issue, it's dropped cleanly.
  • "Teaching" is the default role_type for backfilled rows because that's what 95%+ of basic-school staff records have. Admins can update the type post-migration if needed (e.g. a school's headteacher record gets manually retagged as "Leadership").
  • The original scope field carries the freeform role description ("Class 3 Teacher", "P5 Maths Teacher", etc.) — preserves the admin's original wording, which is more useful than forcing them into a canonical form.
  • For the build estimate: the migration script is ~30 lines of SQL, the new query layer is ~80 lines of Drizzle code, and the UI changes to existing staff list/detail screens are about 2 days of work. Doable in Sprint 1 comfortably.
