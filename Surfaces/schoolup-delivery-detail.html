Who got the message, and when

When a parent says "I never got the notice," the headmistress needs to know whether Omnischools sent it, whether it was delivered, and at what time. The per-recipient delivery detail page exists for two reasons: operational follow-up on failures, and audit when a parent disputes. Two states: the full table view drilled in from announcements history, and the per-recipient timeline drawer for one specific person.

Why this layout

  • Top-down hierarchy: announcement → stats → failures → table. The headmistress drilling into delivery detail almost always wants to know the same things in the same order: what was sent, how many got it, what failed, who specifically. The page architecture makes that flow inevitable.
  • Message preview card at the top reuses the actual message text in italic gold-bordered blockquote. Same pattern as the WhatsApp template authoring page — the message itself is the anchoring fact. Admin can verify she's looking at the right announcement before drilling deeper.
  • Right side of the message card shows channel-by-channel breakdown in a small bordered card. "WhatsApp · 94 · SMS fallback · 8 · Email · not selected." Tells admin which channels were attempted; emails being "not selected" is honest about what didn't happen.
  • "SMS fallback" framing reflects the actual policy: WhatsApp first because of cost, SMS only when WhatsApp can't reach (no opt-in, expired window, user without WhatsApp). The label tells admin why 8 people got SMS specifically — not because they were chosen for SMS, but because WhatsApp didn't work for them.
  • 6-tile stat strip with hero "Reach so far" in navy/gold. Reach (96/102) is the headline because it answers "did the announcement land?" Reading rate (71/96 = 74%) is secondary; admin cares more about reach than read in most operational scenarios.
  • Stats split into Read / Delivered / Pending / Failed. Four states reflect the WhatsApp lifecycle (Read = saw it, Delivered = on phone, not yet read, Pending = sent but no confirmation yet, Failed = didn't get there). SMS doesn't have "Read" — there's no read receipt for SMS — so SMS recipients show as "Delivered" at best.
  • "Cost so far · GHS 5.20 · 8 SMS · WhatsApp included" stat tile makes cost transparent. WhatsApp messages are part of the school's monthly conversation bundle (effectively free per message); SMS costs GHS 0.05 each. Admin sees that this announcement's marginal cost is GHS 5.20 — meaningful number, useful for budgeting.
  • Failures callout in terra-bordered card sits above the table. "4 messages failed to deliver · 3 invalid phone numbers · 1 recipient blocked the school." Pulls failures up because that's the most common reason admin drilled in. Two action buttons: "Review each →" (drill into each failure) and "Resend via SMS" (terra-primary, the destructive-but-corrective action).
  • "1 recipient blocked the school" is a real WhatsApp failure mode. Sarah Amponsah blocked the school's WhatsApp number — a parent disagreement, perhaps, or just hygiene. The system surfaces this honestly. Resending via SMS is the workaround; the school can't unilaterally restore WhatsApp access.
  • Filter bar uses two filter groups + search. "Show: All / Read / Delivered / Pending / Failed" + "Class: All / JHS 2A / 2B / 2C" + search by name/phone. Failed filter is terra-active when selected — visual signal that the user is in "see only the problems" mode. Two filters active simultaneously narrows aggressively.
  • Recipient table groups rows by status with section dividers. "Failed · 4" / "Pending · 2" / "Delivered · not yet read · 25" / "Read · 71 · most recent first." Failures float to the top regardless of which filter is active because they're what admin needs to act on. Within each group, ordering is by recency.
  • Each row: # / Recipient (avatar + name + child) / Channel pills / Last update / Actions / chevron. Six columns, each earned. The recipient cell shows guardian + the child they're a guardian of — admin's mental model is "messages go to parents, but they're about students." Both names visible.
  • Channel pills are the centrepiece. Each pill: small channel icon (gold S for SMS, WA-green W for WhatsApp) + state dot (green/gold/warn/terra) + state label. One row can have multiple channel pills — Mary Mensah's row shows both WhatsApp Failed AND SMS Invalid because both channels were tried. The denser layout matches the mental model.
  • State dots use the locked colour vocabulary: green delivered/read, gold sent (in transit), warn queued, terra failed. Read state has a subtle ring around the dot to distinguish from plain delivered — same dot, slightly more emphasis. Visual signal that someone actually saw it, not just that it landed on a phone.
  • "Read 4:32 PM" inline in the channel pill for read messages. Time-of-read is operationally useful — admin can see a parent read at 4:32 and called the school at 4:45 (the 13-min gap is meaningful). Time embedded in the pill prevents the table from getting wider with another column.
  • Failed rows have terra-tinted background across the entire row width. Hover deepens the tint. Visual scan reveals failures instantly; admin's eye lands on the failed rows even when scrolling through 102 entries.
  • Action buttons differ by failure type. Mary's row (invalid phone): "Update phone" (primary, gold) + "Resend" (secondary). Sarah's row (blocked): "Send via SMS" only. Patrick's row (invalid): "Update phone" + "Resend." Each failure type gets the right action — updating a phone is meaningless if the user blocked you; resending to a blocked user just fails again.
  • Last-update column shows precise time + relative phrase. "2:14 PM" in mono on top, "~3h ago · failed" italic below. Two formats serve two needs: admin verifying the timeline wants the precise time; admin scanning for "what's recent" wants the relative phrase.
  • Chevron column on the far right indicates the row is clickable into a drawer (next state). Convention from the inbox and follow-up flows — clickable rows have a chevron, non-clickable don't.
  • "Resend to failed →" CTA in the page header is the highest-frequency action on this page. Admin drills in, sees the failures, clicks resend. The action is in two places (page header + failures callout) because both are reasonable entry points.
  • "Export CSV" CTA for the audit case. When a parent disputes ("I never got the message"), admin exports the full delivery record and can attach it to the response. The export includes timestamps to the second, channel attempted, channel result, failure reason — the audit trail is the export, not the screen.
  • What's not on the table: No phone numbers visible (privacy + screen real estate). No message content (it's at the top of the page, doesn't need repeating per row). No bulk-select checkboxes (would clutter; bulk action is "Resend to failed" which operates on the failed-filter view, not row selection).
  • What's also not here: No re-target option ("send to All JHS 2 again with a different message"). That belongs in the announcements compose flow, not in delivery detail. Mixing the two would create a wormhole.

Why this layout

  • The drawer slides over the page, not replacing it. Background page is dimmed but visible — admin keeps the context of the broader announcement (102 recipients, sent at 2:14 PM, etc.) while focusing on one specific person. Closing the drawer returns to the table; no navigation lost.
  • Drawer head with recipient identity + delivery banner. Big avatar, "Mary Mensah" in display Fraunces, guardian-of relationship and student ID. Below: terra-tinted banner reading "Did not receive this announcement · Both WhatsApp and SMS failed · Mary has not been informed of the fee reminder." Plain English summary at the top; admin doesn't need to read the timeline to know the bottom line.
  • Banner state changes by recipient. For a successful recipient (Patricia Boateng who read at 4:32 PM), the banner would be green: "Read this announcement · Read on WhatsApp at 4:32 PM, ~50 min after sending." For pending: gold "Sent, awaiting confirmation." The banner answers the headmistress's first question ("did this person get the message?") in one sentence.
  • "Channels attempted" section uses two cards side by side for WhatsApp and SMS. Each card shows: channel icon, name, state text in colour, destination phone number. Both cards in failed-tinted bg because both failed for Mary; this would be mixed (e.g., WhatsApp failed, SMS delivered) for other recipients.
  • Timeline is the centerpiece of the drawer. Vertical rail with circles connecting events. Each circle's colour matches the locked vocabulary: gold for sent/transit, terra for failed, green for delivered/read. Six events shown for Mary's timeline: queued → WhatsApp sent → WhatsApp failed → auto-fallback triggered → SMS sent → SMS failed.
  • Timeline shows precise timestamps to the second. "2:14:03 PM · 2:14:08 PM · 2:14:11 PM · 2:14:14 PM · 2:14:18 PM" — the entire round trip happened in 15 seconds. Second-level precision matters for audit; "around 2 PM" wouldn't be enough if a parent disputes specifics.
  • "WhatsApp template fee_reminder_v2 dispatched" shows the actual template used. Connects this delivery to the WhatsApp template authoring file we designed earlier — the same template name appears here. Visible identifier connects the surfaces; admin who wonders "which template was sent?" can trace it.
  • Failure reason blocks are detailed but readable. When WhatsApp failed: "Meta returned: recipient phone number not registered on WhatsApp." Plain-English explanation: "This is common when a guardian shares a feature phone or recently switched numbers." Technical reference: "META-API · code 131026 · Receiver is unable to receive message." Three layers serve three audiences — the admin understanding the failure, the support agent helping admin, the engineer debugging.
  • Auto-fallback event is its own timeline entry. "Auto-fallback to SMS triggered" with circular ↻ icon. Makes the system's behaviour visible; admin sees that Omnischools tried SMS automatically when WhatsApp failed, didn't have to be told. Trust through transparency.
  • SMS failure has a different explanation than WhatsApp failure. "MTN Ghana's network rejected the SMS." Three possibilities listed: number disconnected, guardian switched networks, number entered incorrectly. Specific to the failure mode — wouldn't be useful to give the same generic explanation for both.
  • "What to do next" section gives actionable guidance. Gold-bordered card with headline "Update Mary's phone number" and a recommendation to ask Mr. Mensah (James's class teacher) for a newer number. System suggests action without taking it; admin still decides. The mention of Mr. Mensah is helpful — class teachers often have updated parent contacts because they communicate with families daily.
  • Drawer footer: 2 actions + recipient counter. Left: "3 of 4 failed recipients" — admin knows there are more failures to review. Right: "Open Mary's record →" (navigate to Mary's contact) + "Resend after update" (primary, gold — the actionable path). Two actions that both make sense; not a choice between right and wrong.
  • Background dimming uses 0.4 opacity, not full opacity 0. The admin can see roughly what's behind the drawer (the message preview, the table) but it's clearly demoted. Modal-like focus without losing context; closing returns to the same scroll position.
  • What's not in the drawer: No "edit message and resend" option (re-targeting belongs in compose). No "block this guardian" action (out of scope). No conversation history with this guardian (separate to delivery detail; lives in Mary's contact record). No detailed read-receipt history if she'd read the message twice (WhatsApp doesn't expose multiple reads).
  • What's also not here: No carrier billing detail. The cost (GHS 0.05 SMS) is shown at the page level, not per-recipient — granular cost attribution is more detail than admin needs at the recipient level.
