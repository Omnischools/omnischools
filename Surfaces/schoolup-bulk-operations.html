Click once, change many

Bulk operations as a system-wide pattern. Schools handle hundreds of students; clicking through one record at a time when the same change applies to 47 of them is the kind of friction that makes admin staff abandon software. Selection becomes a first-class object. Demonstrated here on the student list — the same multi-select toolbar, sticky selection bar, confirm-with-preview flow, and post-action result card apply to billing, attendance, communications, and staff lists. One pattern, every list.

Selection mechanics

  • Master checkbox in the table header has 3 states: empty / checked / indeterminate (gold dash). When some rows are selected, the master shows indeterminate — clicking it deselects all. Standard semantics; matches user expectations from email and spreadsheet apps.
  • Selected rows have gold-tinted background matching the Selection Bar at the bottom. The visual ties the selection state to the action surface — admin sees what they've chosen and the action affordance simultaneously without moving their eyes far.
  • Selection persists across pagination and filter changes. If admin selects 47 students, then filters to JHS 2 to verify, the count stays at 47 and the bar still shows "47 of 312." Selection is a property of the session, not of the visible page; this is how schools actually work — they want to see "is everyone in my selection in JHS 2?" without losing the selection.
  • Withdrawn students show "— closed —" with subtle balance; selecting a withdrawn student is technically possible but most bulk actions skip them or warn. The skip-warning step in the confirm flow (next region) catches this.
  • Selection bar is the most consequential UI element: sticky to the bottom (position:sticky), navy background to match the sidebar (visual continuity with system chrome), gold accents for the count and primary action. The bar appears the moment any row is selected and disappears when none are; it's not always-visible chrome.
  • Selection bar has 3 zones: count + master checkbox (left) / smart helpers (centre) / actions (right). The smart helpers row is the pattern's signature: based on what's selected, the system suggests adjacent selections — "All overdue families" (the apparent intent of the current 47-row selection in gold), "Page (8)" (just the visible page), "JHS 2 only (15)" (intersect with current filter), "All 312" (everything).
  • Smart helpers are inferred: the system noticed that 47 of the selected rows have the same property (overdue), so it's offering "All overdue families" as a one-click superset/lock-in. If admin meant "all overdue" but missed a couple, this lets them recover without manually scrolling.
  • Action buttons use the intent vocabulary: "Send message ▾" (with dropdown for channel), "Update class ▾", "More ▾" (catch-all for role-related actions), and the primary "Apply discount →" in gold. The primary suggested action is contextual to selection content: 47 students all with overdue balances surfaces "Apply discount" as the high-confidence guess.
  • Clear selection is a small italic gold link, not a styled button — it's deliberately less visually weighted than the action buttons. The path of least resistance is to act, not to clear; admin chose these rows on purpose.
  • Count uses Fraunces italic gold "47" in a 20px display setting — visually echoes the editorial cards elsewhere. The count is the headline of the action; 47 is a meaningful number, not a database statistic.
  • "304 more students · scroll to load · 40 more selected below the visible window" · the table foot tells admin that selection extends beyond the visible page. This is what makes "select all overdue" trustable; admin can verify the count without hunting through pages.

Confirm-before-commit principles

  • Bulk actions never apply on the first click. The selection bar's gold "Apply discount →" opens this confirm sheet · admin must walk through the 4 steps before the action commits. The action is reversible-via-cancel until the final button; mistakes are caught before they're recorded.
  • Header restates the scope: "Apply discount to 47 students." Then meta tells admin the consequences in plain English: "47 audit log entries grouped under one bulk event · receipts regenerated · parents notified within 5 minutes." Side effects are visible upfront; admin doesn't discover them after the fact.
  • Step 1 picks the action variant. Discount can be fixed amount or percentage — same shape, different math. The selection determines step 2's input; if percentage chosen, step 2 swaps to a percent input.
  • Step 2 captures parameters with the audit-log labels visible: "Reason · Visible in audit log · printed on each receipt · parents see this on their invoice." The reason field is required (no anonymous bulk discounts) and admin sees exactly where it appears.
  • "Effective: Immediately on confirm · 22 Oct 2024 · 2:18 PM" with help text "forward-only." Locks in the temporal semantics; bulk discount doesn't retroactively change paid receipts, only outstanding balances. Already-paid amounts are not refunded by this action.
  • Step 3 preview shows the actual diff per row: "GHS 340.00 → GHS 290.00" with terra strike-through on old, green bold on new. Admin sees exactly what the changes will be before they happen; the preview is the dress rehearsal.
  • Skip warnings in warn-tinted card: "2 of the 47 selected students will be skipped." Lists each skipped student by name with reason. The system makes the safe call: Sandra Osei has zero balance (discount would create credit, which is a different operation); Joseph Asante's invoice is under review (locked by Joyce). Admin sees this before committing and can choose to investigate or proceed.
  • Step 4 is the consent gate: terra-tinted card with explicit consequence statement. "Discounts are forward-only and cannot be reversed automatically · undoing requires manually adding GHS 50 back to each student's invoice" · admin understands the asymmetry before committing.
  • Commit footer summarises the final action in editorial Fraunces: "Reducing 45 of 47 balances by GHS 50 each." Note: 45, not 47 — the count adjusted for the 2 skipped students. The number admin commits to matches what actually happens.
  • 3 footer buttons: neutral Cancel / neutral Save as draft / gold Apply (the commit). Save as draft is a deliberate option — admin can prepare a bulk action and have a co-admin review the preview before applying. Useful for sensitive ops at multi-admin schools (file links to the activation-segregation pattern from earlier).
  • Consistent confirm-flow vocabulary across modules: this same 4-step shape (Choose action / Set parameters / Preview changes / Acknowledge & commit) applies to bulk class promotion, bulk attendance correction, bulk SMS send. Once admin learns the rhythm, every bulk operation feels familiar.

The result card

  • Result card opens in place of the confirm sheet immediately after commit. No modal, no navigation away; admin sees the outcome of their action in the same surface they configured it.
  • Big green checkmark ring + editorial header: "Discount applied to 45 students." One sentence answers "did it work?" · meta line tells the secondary story: "2 skipped · 45 receipts · 45 notifications · 1 audit event · sealing at midnight."
  • 4-stat grid summarises the action: Applied (45 green) · Skipped (2 warn) · Total reduced (GHS 2,250) · Parents notified (45 with channel breakdown). The accountancy of the bulk action is visible at a glance; this is what admin will refer back to next week when reconciling.
  • Per-student outcome rows show the actual change: "340.00 → 290.00" with green checkmark ic. The 2 skipped rows are warn-tinted with ⊘ icon and italic "no change" — clearly distinguishable from the successes. Same diff format as the confirm preview; admin recognises the visual continuity.
  • Receipt numbers attached to each outcome: "JHS 2A · receipt #2219." The new receipts are a tangible byproduct; admin can drill from any row into the regenerated PDF if a parent disputes.
  • Gold-tinted audit link footer: "This action created audit event ev_2024_10_22_14_18_103 · grouped 45 invoice updates + 45 receipt regenerations + 45 parent notifications as one bulk record." Direct link to the audit-log surface from file 1 of this batch · the bulk row in the audit feed is the same event referenced here.
  • "all visible to other admins immediately" · transparency is built into the action. Joyce will see Florence's bulk discount in her audit feed within 30 seconds (the auto-refresh interval); no need for a separate notification — the audit log is the notification channel.
  • Header has 2 actions: "View audit entry →" (jumps to the audit drill panel for this event) and "Download receipts" (zip of all 45 PDFs). The two next steps after a bulk action: review the record, or distribute the artefacts.
  • Skipped rows preserve the reason: "Sandra Osei · already GHS 0 balance" / "Joseph Asante · invoice under review." The skip reason matches the warning shown in the confirm step — admin knew this was going to happen and the result confirms the system did what it said.
