Your data, your school's

Schools should be able to walk away with all their data — that's table stakes for any SaaS, but particularly important when the MoE asks "what happens if Omnischools shuts down?" Three flavours of export: quick one-tap downloads for routine reporting (this month's attendance, this term's billing), a custom builder for ad-hoc queries, and a full data archive for portability. The archive flow is the trust signal — encrypted ZIP, server-compiled, emailed when ready.

Why this layout

  • Three flavours of export, one surface: quick exports (one-tap pre-shaped) / custom builder (3-step wizard) / full archive (encrypted ZIP, slow, server-compiled). Different cognitive modes: quick is "I know what I want, give it to me" · custom is "I want a slice that doesn't exist as a preset" · archive is "I want everything." Three regions on one page so admin doesn't have to learn three different surfaces.
  • Trust banner in gold-tinted card at the top: "Your data is portable by design. Every export is generated fresh from your live records — no stale snapshots · matches Ghana DPA 2012 subject access standards · suitable for MoE inspections, GES returns, GRA audits, and provider transitions · never billed extra for export." Establishes the four guarantees upfront: freshness, regulatory adequacy, multi-purpose, no paywall.
  • "Never billed extra for export" is the key trust phrase · many SaaS providers gate exports behind premium tiers or per-export fees. Omnischools's commitment is structural: data export is free regardless of plan · this is how MoE pitch credibility is built.
  • Quick exports as 6-card grid with consistent shape: ico (module-coloured) + format tags (CSV/PDF) + name in italic Fraunces + descriptive line + meta row with last-downloaded + DOWNLOAD link. Same card shape for every quick export; admin learns the pattern once.
  • "Last downloaded · 3 days ago" / "Never downloaded" shown on every quick export · signals freshness expectation. If the same export was downloaded yesterday it's the daily routine; if never, admin discovers the option.
  • 6 quick exports cover the most common needs: attendance / billing / parent contacts / student roster / gradebook / SMS+WA logs. Each is a distinct operational moment for the admin: monthly reporting, term-end billing review, emergency contact backup, roster verification, report card prep, communication audit.
  • Custom builder card with gold border lives in the same screen as quick exports · 3 steps: entities / fields & filters / format. Same builder pattern as reminder series; admin already knows how this works · no new mental model.
  • Builder example is "JHS 2 students with low attendance" — a real ad-hoc query Florence might build for a parent-teacher review meeting. Concrete enough to demonstrate the value; the kind of query no preset would cover.
  • Entity grid shows record counts: "Students 312 records" / "Attendance 26,520 records" / "Audit log 1,287 events." Admin sees the scale of what they're querying before committing · prevents accidental million-row queries.
  • Field list groups by entity with checkboxes · PII fields (home address, parent phone) are visible but flagged "PII · approval needed" in italic gold. Sensitive data is gated; can be selected but requires elevated permission to actually export · audit-logged when used.
  • Filter rows in field list shown as gold-bold rows: "Filter: % < 80" and "Filter: class = JHS 2A/B/C". Filters are first-class fields; admin sees them in the same list as the columns being included · the query is one cohesive object.
  • Format selector with 4 options: CSV (selected, "most flexible") / Excel (multi-sheet) / PDF (printable) / JSON (machine-readable). Each format has a 1-line characterisation · CSV is the safest default but Excel is offered for users who want multi-sheet structure without exporting and re-importing.
  • Preview card narrates the result before commit: "Result: ~14 rows · JHS 2 students with attendance below 80% · 9 fields per row." Admin sees what they're going to get before generating · same preview-before-commit principle as bulk operations.
  • "Excludes: Phone numbers, home address (PII · would require approval)" visible on the preview · privacy-by-default is shown as deliberate, not hidden · admin sees that they could request approval if they really needed PII.
  • Save query option in the save bar · "Save query only" lets admin define a custom export shape without running it · gets added to the quick exports grid for one-tap re-use later. Custom queries become quick exports over time; the system gets smarter as the school works with it.
  • Full archive card in dramatic navy/gold · the visual emphasis matches the operational gravity. "All your school's data — in one encrypted ZIP" · meta narrates: server-side, 15—30 min compile, secure download link, expires in 7 days, password-protected with admin-chosen key.
  • Three commitments shown as gold-checkmark pills on the archive card: "Open formats only — CSV, JSON, PDF" / "Includes audit trail" / "No Omnischools lock-in." Anti-lock-in stated explicitly; this is a competitive/regulatory differentiator and Omnischools owns it loudly.
  • Right side stats panel: estimated size (~38 MB) / records included (~36,200) / last requested (Never). Concrete scale before committing; admin knows what they're asking for · "Never" reinforces that this is a rare/significant request.
  • Request flow is async with email delivery: not an instant download. Acknowledges archive generation is slow; better to wait 15 min for a complete archive than to hang the browser. The email link with 7-day expiry + password protection is the secure-handoff pattern.
  • Recent exports table in section 04 · 6 columns: Export name / Format / Actor / When / Size / Re-download. Every export is logged in audit · table is filterable surface of those audit entries · admin can see who downloaded what when · cross-references with audit log file 1.
  • Bulk transcript generation appears in the recent exports table · cross-file linkage with bulk operations file. Bulk ops that produce files automatically appear here; export and bulk are connected, not separate worlds.
  • "Re-download" link for past exports rather than regenerate · the original file is preserved for a configurable retention window (90 days default) so admin can re-fetch without rebuilding. Saves time + cost; especially useful for the slow archive flow.
