Why this layout

  • Send identification card at the top with module-specific icon + colour (R for reminder, gold for announcement, G for GES statutory). Shows: "Send #3 · Friday 17 October" · meta line with audience size, template, cost. The same screen header shape works for any broadcast type; only the icon and module-context line change per source.
  • Module context note in gold-tinted card says explicitly: "Same delivery-detail surface opens for any broadcast — an announcement, a reminder series fire, an SMS reminder from billing, a statutory comm from GES." Tells admin upfront that this UI is generalised; muscle memory transfers across modules.
  • Hero row leads with the editorial summary: "43 of 47 reached" · "91.5% delivered · 28 read confirmed · 2 failed · 2 pending · 5 parents replied." One sentence for the whole send. Then 4 small tiles by state in their colour vocabulary (green Read 28, neutral Delivered 15, warn Pending 2, terra Failed 2).
  • Channel breakdown strip with stacked bars per channel. WhatsApp: 65% read + 23% delivered + 12% failed → SMS. SMS: 79% delivered + 14% pending + 7% invalid. Auto-fallback footer note: "fired for 4 recipients when WhatsApp wasn't deliverable." Schools can see the resilience of the multi-channel design at a glance.
  • Email channel shown as "Not used" with italic Fraunces footnote: "parents prefer WA + SMS for reminders." The strip lists all channels even when one didn't fire; admin sees the full picture, including the deliberate design choice that emails aren't appropriate for short reminders.
  • In-app row shows "28 of 47 parent accounts active in app · receive copy" — the cross-channel surface includes the app delivery as an additional path. Parents who use the app see all sent communications there too; not a substitute for SMS/WA but a parallel channel.
  • 6 filter pills with counts: All / Read / Delivered / Pending (warn-tinted) / Failed (terra-tinted) / Replied. Plus a search box. Failures are filterable to the top by colour, helping Florence go straight to the recipients needing intervention.
  • Per-recipient table sorted by delivery state with failures first. Failures and pending get terra/warn row treatment; reads, delivered states scroll below. The most operationally important rows are at the top.
  • 5-column table: Recipient / Channel attempts / Final state / Delivery timeline / Action. Same structure that works for any send type. Channel attempts column shows the auto-fallback chain when relevant: "WA · failed → SMS · sent" with note "Auto-fallback after WA delivery failure."
  • Channel attempt pills include a "n/a" state for parents not on WhatsApp at all (so the system didn't try that channel) — distinct from failed (system tried and the channel rejected). Joseph Asante shows "WA · n/a" → "SMS · failed" because he's never been on WA. Honest distinction between "didn't try" and "tried and failed".
  • Mini timeline column in mono font shows the exact event sequence per recipient: "9:00 WA sent / 9:00 ✓ delivered / 11:42 ✓✓ read." Compressed audit trail visible at the row level; full conversation drill-through via the row action.
  • Per-row action button is contextual: Failed rows get terra "Update contact →" (the operational fix). Pending rows get neutral "Resend →." Successful rows get gold "View thread →" (drill into the parent conversation history from file 1). Opt-out gets a "Note opt-out" record-keeping action.
  • Patricia Boateng's row demonstrates cross-file linkage: this delivery-detail row shows her receiving the 17 Oct SMS reminder, and the timeline notes "Replied via WA on Wed · 5 days later." Clicking "View thread →" opens her conversation history (file 1) with the Wed 22 Oct messages visible. Three files form one continuous narrative; the same data viewed from three perspectives.
  • Samuel Owusu-Ampofo opt-out row shows "Read 9:08 AM · opted out for next time" — admin sees that one parent specifically requested no more of this series. Opt-outs are tracked at the per-recipient level, and the next reminder-series fire automatically excludes them. The action button is "Note opt-out" rather than "View thread."
  • Sandra Osei's row in terra shows the worst-case state: "WA · failed → SMS · failed" with metadata "Both numbers invalid · email not on file." The system tried every channel; the operational answer is "Update contact →" (not "Resend"). Distinguishes intervention from retry.
  • "Sorted by delivery state · failures first" meta on the toolbar tells admin the ordering principle. The default sort puts the most actionable rows first; admin can re-sort if she wants chronological or alphabetical. The default is opinionated: failures matter most.
  • The send card's actions row offers 3 buttons: "View series →" (back-link to the series this fire belongs to) · "Audit log" (full system event trail) · "Send manual follow-up" (primary action for following up with the failures). Three intents at the broadcast level: navigate, inspect, escalate.
