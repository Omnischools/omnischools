Why this layout

  • Message summary at the top in two columns. Left side shows the actual message text in a gold-accented quote block — admin reads exactly what was sent before reading delivery stats. Right side shows the operational metadata (when, by whom, audience, total cost, reference ID). Important to anchor the page to the message itself, not just numbers.
  • Resend banner is the load-bearing call-to-action. "72 parents didn't receive this message" with the resend cost (GHS 3.60) and the Gold "Resend to 72 failed →" button. Most visits to this page end here — admin sees the headline, taps resend, done. The banner makes that path one click from page load.
  • Resend explicitly states it won't double-charge. "Only failed numbers; delivered recipients won't be charged again." Removes the "wait, will this re-bill me for the 528 already-delivered?" question that would otherwise stop admin from clicking.
  • Featured KPI is "Total reach" in navy with both the absolute count (528) and percentage (88%). The channel split bar below it shows the composition: 84 WhatsApp + 372 SMS + 72 in-app + 72 failed = 600. Visual ratio of how the message reached people.
  • "Delivered, not yet read" is its own KPI, separate from confirmed-read. SMS has no read receipts, so 444 SMS recipients can never have a "Read" status. Conflating them with WhatsApp/app would be misleading. The card explicitly notes "SMS has no read receipts."
  • Failure reasons grouped into 4 categories. "Number unreachable" (48, retryable), "Invalid number" (14, needs phone update), "SMS opted out" (7, parent's choice), "Carrier rejected" (3, retry). Each tile shows the count and a one-line interpretation. Admin reads the breakdown and knows which failures are recoverable.
  • Different per-row actions based on failure type. Number unreachable / carrier rejected → "Retry" button (sends to that one recipient). Invalid number → "Update phone" (jumps to guardian record). Opted out → "View" (read-only; can't override parent's STOP). The action matches the recovery path; admin doesn't have to reason about it.
  • "Failed" filter pill in red, active by default when the page loads from a low-delivery announcement. The page's primary purpose is failure investigation; pre-filtering matches the use case.
  • Channel pills use the brand colours. SMS in gold, WhatsApp in green, In-app in neutral. Visual hierarchy mirrors the channel hierarchy — gold for the universal-reach workhorse, green for the cheaper richer modern option, neutral for the supplementary push.
  • Status icons are channel-aware. WhatsApp uses the double-check (✓✓) in WhatsApp blue for "Read" — matches what parents see in the actual app. In-app uses ⊕ for "Tapped" — different mental model from "read." SMS just gets a single ✓ for "Delivered" because that's all SMS gateways report.
  • "Read at 12:08 PM" / "no read receipt for SMS" in the When column gives temporal detail where it exists, transparent admission where it doesn't. Avoids misleading admin into thinking SMS recipients haven't read when really we just don't know.
  • Successful deliveries shown as a small sample, not the full 528. Admin who wants to see all delivered recipients clicks the filter pill. Default view privileges the actionable (failures) over the routine (successes).
  • "Opted out" gets a soft treatment. Marked as failed in the count, but the action is "View" not "Retry" — admin shouldn't pressure-resend to a parent who explicitly opted out of SMS. Includes the date they opted out so admin sees this is intentional, not a system error.
