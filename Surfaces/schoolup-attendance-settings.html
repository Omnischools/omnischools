Why this layout

  • Three regions, vertically stacked, in priority of how often they get touched. Calendar at the top because it changes every term; daily schedule in the middle because it's set once per school; alert rules at the bottom because they're the policy decisions that ripple through everything else but rarely change. The page architecture itself reflects how often each region needs attention.
  • Term selector at the top: 3 pills. "Term 3 2024/25 · read-only" / "Term 1 2025/26 · current" (active) / "Term 2 2025/26 · draft." Past terms are auditable but not editable; current is editable; future term is a draft. Schools plan ahead — the headmistress draws up Term 2's calendar in November, knowing she'll switch to it on Day 1 of January.
  • Default notice in gold-tinted card at the top of the body. "These are the Omnischools recommended defaults. We've calibrated them for typical Ghanaian basic schools — change them only if you have a reason." Permission to leave things alone. A non-confident admin can read this, decide they don't need to change anything, and close the page. The "Reset to defaults" button on the right gives admins who've broken something a way back.
  • Region 1 — Term boundary strip in 4 cells: Term begins / Term ends / Days marked / Holidays noted. The first two are editable date fields; the last two are computed read-only. Boundaries vs progress vs config all visible on the same horizontal strip — admin sees the term as a temporal whole.
  • "Days marked: 47 of 62 · 76% through term" isn't editable but contextualises the calendar. Admin reading this knows the term is 76% done and 15 school days remain. Helps with planning ("can I add a sports day in week 13?" — well, that's exam week, so no).
  • Calendar visual is the centrepiece of Region 1 — week strips with day cells. Same vocabulary as the term grid in the desktop teacher view: green school days, dashed weekend tiles, hatched holiday days, gold exam days, dashed-border future days. Visual continuity means admin already knows how to read this; it's the same drawing they see everywhere else.
  • Holidays/events appear as full-width banners between week rows. "Founders' Day · public holiday · school closed" / "Mid-term break · 5 school days · whole week off" / "Sports Day · school event · attendance auto-marked" / "End-of-term exams · 5 days · attendance still tracked." Each banner has a name in italic Fraunces gold + a one-line plain English explanation of what it means operationally.
  • Banner text answers "what does this mean for attendance?" Founders' Day → school closed (no register). Mid-term break → no register, no SMS. Sports Day → register is auto-marked Present for everyone (excursion mode). End-of-term exams → registers continue as normal. The banners aren't decorative; they're operational guidance.
  • Calendar action bar at the bottom: 4 buttons. Add holiday / Add school event / Add exam week / Import GES public holidays. The fourth button is the most important — most public holidays are gazetted nationally, so importing from GES saves the headmistress from typing them in by hand. The system pulls them and she just confirms.
  • Holidays list below the calendar visual in chronological order. Each row: date in mono caps / name in display Fraunces / type tag (Public holiday / School-set / School event / Exam week) / edit + delete actions. The list is the editable record; the visual above is the readable rendering. Both views of the same data.
  • Type tag colour-coding: Public holiday in neutral grey (not changeable, set by national government), School-set in gold-tinted (school decision), School event in gold-bg (operational), Exam week in saturated gold (special). Visual hierarchy matches editorial control — public holidays are external authority, others are school choices.
  • Region 2 — "A typical day at Christ the King" headline in display Fraunces. Personalises the configuration. Admin reading this isn't editing abstract settings; she's describing how her school works.
  • Time bar with three markers: 8:00 AM (green), 8:15 AM (gold), 3:00 PM (terra). Same colour logic as the rest of the system — green for "all good," gold for "late," terra for "ended." Admin can drag markers or edit time inputs below. The visual is teaching: the colours show that "late" is in the gold zone (between school-start and end-of-school), not before or after.
  • Hour axis at the bottom (6 AM → 6 PM) in mono. Admin who wants to add a new marker (e.g., "afternoon registration at 1 PM") can position it. The axis shows the temporal scale clearly.
  • Three time inputs below the bar with each one paired to a help text. "Time before which students must arrive to be marked Present" / "Students arriving after this are marked Late instead of Present. 15 minutes grace period from start" / "Used for the daily 3 PM auto-SMS that goes to absent students' guardians." Each setting explains its consequence, not just its value.
  • Edit window stepper in its own card. "Register edit window · 24 hours" with stepper buttons. The stepper is gold for the value (24 hours) — same convention as the threshold rules, same visual idiom of "this is the tunable thing." The card stands alone because edit window is conceptually separate from time-of-day settings.
  • Region 3 — Notifications & alerts split into 3 cards. SMS to parents (left) / Pattern detection (right) / Daily digest (full-width below). Each card has an eyebrow + display headline + descriptive lede + the actual controls. Headlines tell the story: "When to auto-message guardians" / "When the system flags a student" / "When and where you'll get the morning briefing."
  • SMS toggles show real cost figures inline: "~20 students/day on average · GHS 1 / day" / "would add ~3 students/day · GHS 0.15 / day" / "would add ~290 students/day · GHS 14.50 / day." Cost transparency for honest decision-making — admin considering reassurance SMS sees the GHS 14.50/day figure (~ GHS 90/week, ~ GHS 1,200/term) and can decide if it's worth it.
  • Description language acknowledges convention. "Most schools leave this off — late arrivals are usually known by guardians." Soft guidance rather than hard "we recommend off." Admin who has a reason to enable it isn't being lectured; admin who's unsure has a default to fall back on.
  • Toggle pills are simple on/off, not multi-state. Green when on, grey when off, with the dot sliding. No tri-state, no "send only on weekdays," no "send to specific parents." Complexity is in the rules, not the toggles. If admins need conditional SMS, that's a future feature.
  • Pattern detection card replays the threshold rules from the alerts file but with steppers. Each rule is a row: severity pill / plain-text rule / stepper for the threshold. Same rules everyone has seen, now editable. Admin can lower "Long absence: Watching" from 3 days to 2 days if their school wants tighter detection.
  • Mini-stepper with gold italic value ("3 d", "70%", "4/6"). Same Fraunces italic gold as everywhere else for the user-tunable parameter. Visual signature: italic gold = "this is the lever you can pull."
  • Digest preferences card with three channels: Email / WhatsApp / In-app feed. Each row has icon + channel name + detail (recipient address, cost note) + time input + toggle. Email is on by default at 07:30 AM; WhatsApp is off (extra cost); in-app feed is forced-on (greyed toggle, "always on"). The forced-on state is honest — bell-icon notifications are core, not optional.
  • "Counts as 1 conversation per day" on WhatsApp digest reminds admin that WhatsApp Business has conversation pricing. Enabling daily WhatsApp digest = ~30 conversations/month. Headmistress sees the operational consequence, not just the toggle.
  • Sticky save bar at the bottom with "3 unsaved changes" terra pill + a summary of the changes ("Edit window: 24h → 48h · Late SMS: off → on · Below 70% threshold: 70% → 75%"). The summary is precise — admin sees exactly what's about to change before clicking Save. Discard reverts. No silent auto-save on settings this consequential.
  • The summary line uses arrows (→) to show before-after. "Edit window: 24h → 48h" reads as "was 24, will become 48." Same pattern as the diff in the edit-request approval drawer. Visual consistency across the system: anywhere a value is changing, it shows old → new.
  • What's not on this page: staff role configuration (separate Settings tab), branding/logo/stamp upload (separate tab), billing settings (separate tab), SMS account credentials (separate tab), GES integration (separate tab). This page is about attendance only; the broader Settings has its own architecture. The tabs at the top of the head section show the full Settings landscape.
  • What's also not here: no "schedule different days for different classes." All classes share the term calendar. No "different SMS rules for different classes" — school-wide policy. No "vary thresholds by year group" — uniform application. MVP1 keeps configuration simple; differentiation is a future iteration.
