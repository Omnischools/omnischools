Submit to Meta

Omnischools submits the template to Meta on your behalf. You'll be notified when the status changes — approved, pending, or rejected with feedback.

Why this layout

  • Five-section composer in one page, not a wizard. Templates are document-shaped — header, body, footer, buttons all relate to each other and benefit from being visible together. The numbered steps in card labels keep the linear feel without hiding state.
  • Template name is snake_case mono in the field. "term_report_ready_v1" — versioned because templates can't be edited after submission; v2 means a new template. The mono font and underscore convention match Meta's API identifier expectations and signal "this is technical" to admin.
  • Three-tile category picker. Utility selected by default with "Most used" tag — 80% of school templates are Utility. Marketing is available for term-start announcements, holiday closures. Authentication is disabled with "SMS only" tag — OTPs in Omnischools route through SMS because WhatsApp's 24-hour conversation window doesn't fit short-lived auth codes, and users expect OTPs via SMS in Ghana.
  • Header type as a 4-segment control. None / Text / Image / Document. Document selected — the most powerful for school use. The filename pattern uses placeholders ({student_name}_term_report_{term}.pdf) so the file Akosua's mum receives is named "Akosua_Boateng_term_report_Term_1.pdf" — recognisable, searchable in their WhatsApp.
  • Body editor with a variable insertion bar. Click {parent_name} or {student_name} and the placeholder inserts at cursor position. The variables themselves are styled like syntax tokens — gold-bordered chips — both in the editor and in the preview render. Visual distinction between content and variables.
  • Variable list with sample data is non-negotiable for Meta. "Variables & sample data — Meta requires example values for review." Without realistic samples, Meta can't tell if the template makes sense; rejections happen. Each variable shows where its value comes from in production ("from guardian record", "from school settings · always full name").
  • School name variable annotation: "from school settings · always full name." This is the explicit confirmation that suffix-stripping doesn't apply to WhatsApp templates. The placeholder always resolves to "Christ the King JHS" in full.
  • Footer optional and limited to 60 chars. "Reply STOP to opt out of WhatsApp updates" is the standard pattern — both compliance hygiene and a real user benefit. Surfacing the opt-out path matters under Meta policy.
  • Up to 3 buttons with three button types. URL (deep link, can include variables for personalisation), phone call (one-tap to school office), quick reply (parent taps to send a pre-set response back). The example uses URL + phone — appropriate for a "report ready" notification.
  • WhatsApp preview is design-faithful. Real WhatsApp visual conventions: avatar with verified checkmark, business name with green tick, document attachment block, message body with variables rendered in highlighted chips, footer in small grey, action buttons in WhatsApp blue with the underline that the real app uses. Two ticks in light blue (read receipts).
  • Verified checkmark next to school name — the real WhatsApp Business verification badge. Schools with this badge get higher trust from parents (matches the "from a real bank" feeling). Important enough to render in the preview accurately.
  • Validation card live-updates. Six green-check items that pass; one warn item flagging that document headers + 2+ buttons trigger Meta's manual review queue. The warn isn't a problem to fix — it's an expectation calibration. Some templates auto-approve in minutes; this one will probably take a day or two. Telling the admin upfront prevents the "why isn't this approved yet" anxiety.
  • "Submit to Meta" button is WhatsApp-green. Anchors the action in the partner brand. Clicking submits via Meta's API; admin sees the status page next.

Why this layout

  • Three states stacked on one page shows the full lifecycle. In production, only the current state would render — but seeing all three together helps validate that each state is informative and actionable.
  • Status hero in different gradient backgrounds. Approved fades from green to white; pending from gold to white; rejected from terra to white. The colour wash sets the emotional tone before any text is read.
  • Status icon in a 56px circle — checkmark for approved, dots for pending, × for rejected. Iconographic at any zoom level. The size is deliberate: the page is communicating a state change, not just data.
  • Right-side meta column shows submission timestamps, category, and contextual data (cost per use for approved, queue position for pending, suggested category for rejected). Compact key-value pairs in Fraunces — the answers an auditor or admin would ask in sequence.
  • Step timeline below the hero visualises progress through Meta's review pipeline. Drafted → Submitted → Auto-checks → Manual review → Approved/Rejected. Done steps in green; current step pulses gold; future steps grey. Clear at-a-glance status without reading any text.
  • Approved state shows production readiness — "237 opted-in parents are eligible to receive messages from this template." Connects the abstract approval to a concrete operational fact; admin sees that the template now does work.
  • Pending state has a queue position estimate ("~3-7 hours"). Even an approximate figure helps admin plan. If the template is needed for an imminent send, admin knows whether to wait or fall back to SMS.
  • Rejected state quotes the actual reason in the rejection text, not a paraphrase. "Template categorised as Utility but body contains promotional language ('special offer', 'limited time')." Meta's feedback is specific; surfacing it verbatim is more useful than summarising.
  • Two action buttons after rejection. "Edit & resubmit" preserves the body and lets admin revise. "Accept as Marketing" recategorises and resubmits with no body change. This second path is the magic — when the only issue is categorisation, admin can fix it in one click without re-thinking the message.
  • Templates can't be edited post-submission; only duplicated as new versions ("v1" → "v2"). The "Duplicate as new version" header button enforces this convention. Meta's API treats edits as new templates anyway, so the UI matches the underlying reality.
  • Archive button in the header for templates that should no longer be used. Archived templates remain in the audit trail but disappear from the active picker. Schools that retire a template (e.g. switching from quarterly to termly reports) need this without losing history.
