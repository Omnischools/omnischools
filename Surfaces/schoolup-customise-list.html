Build the exact list

Pre-built segments work for most cases ("All overdue families," "JHS 3 only"), but sometimes the right list isn't any of them. The customise list flow lets the headmistress combine filters across class, payment status, aging, and special tags — then refine the result by hand-checking individuals to include or exclude. Filter to narrow, scan to verify, uncheck the ones who shouldn't be reminded.

Why this layout

  • Modal over the send-reminders drawer, not a full-page navigation. Admin entered the customise flow from a specific reminder she's drafting; closing the modal returns her to that drafting context. Maintaining context matters because the reminder content (message text, channel choice) is still being composed in the drawer behind.
  • Background page peek dimmed at 0.4 opacity with a generic light grey card to suggest "the send-reminders drawer is back there." Doesn't try to redraw the drawer perfectly — admin has the mental model of "I clicked this button, here I am, I'll go back when I'm done."
  • Modal head: crumb + "Customise recipient list" + lede "Filter to narrow · uncheck individuals · changes are not saved after this reminder is sent". The "not saved" note is critical — admin understands this is a one-shot list construction, not a saved segment. Sets expectations honestly; a future iteration could add saved segments but MVP1 doesn't.
  • Two-column body: 280px filter rail on the left, recipient list on the right. Filters narrower because they're configuration; the list is the content and deserves the space. Filter rail scrolls independently from the recipient list — admin can scroll filter options without losing the list, and vice versa.
  • Filter rail head explains the starting point: "Start from the school's full 312 students and narrow down. Each filter further refines the list on the right." Helps admin understand the model — filters are subtractive from the school-wide population, not additive from zero.
  • Active filters summary card at the top of the filter rail in gold-bordered tint. Reads in plain English: "JHS 3 · overdue invoices only · aged 31—60 days · transport users excluded." Translates the configuration into a sentence; admin can read this aloud and decide if it's right. "Clear all filters" link in terra at the bottom of the card.
  • Six filter groups, each with its own pattern. Class & year (multi-select pill grid 3×3), Payment status (color-coded status pills), Aging (range slider), Tags (multi-select with negation), Communication (boolean filters), Last reminder (mutually-exclusive time bands). Each group uses the right control for its data shape; not every filter is a pill.
  • Class pills show counts inline: "JHS 3A 34" / "JHS 3B 35" / "JHS 3C 33". Admin sees the population behind each class while choosing. The selected pills fill with navy and italic gold. The unselected pills are surface-coloured.
  • Status pills use the locked colour vocabulary when on: Overdue terra / Partial gold / Paid green / Exempted neutral. Visual reinforcement — clicking "Overdue" highlights with terra to mean "this is a problem state being filtered for." Same colour grammar as everywhere else.
  • Aging is a range slider, not buckets. Two handles let admin pick "31—60 days" or "0—30" or "any custom range." Continuous control for a continuous variable (days). Range readout in italic gold ("31—60 days") matches the system's tunable-parameter visual signature.
  • Aging slider tick marks at 0/30/60/90+ in mono. Common reference points without forcing them as buckets. Admin can drag handles to anywhere; the ticks just orient the eye.
  • Tags filter supports negation. "−Transport users" pill in terra-tinted with a minus prefix means "exclude this tag from the result." Different visual state from "include this tag" (which would be navy on). Inclusion vs exclusion is a real distinction in segmentation; the visual difference makes it explicit.
  • "BECE candidates" / "Scholarship" / "Sibling discount" / "New admission" / "Repeating year" are realistic Ghanaian school tags. Not generic CRM segments — these reflect actual operational categories (BECE = end-of-JHS exam preparation; sibling discount = fee reduction for multiple-children families; repeating year = students who didn't promote). Ghana-school-specific, not generic SaaS.
  • Communication group filters by channel availability: "WhatsApp opted in / Has email on file / Reminded last 7 days." Useful for cost optimisation — if admin filters to "WhatsApp opted in" she'll send via WhatsApp not SMS, saving GHS 0.05 per recipient.
  • Last reminder group prevents over-messaging: "Never / In last 7 days / 8—14 days ago / More than 14 days." Helps admin avoid annoying parents with too-frequent reminders. Could be combined with "Reminded last 7 days" exclusion to send only to people who haven't been reminded recently.
  • Right panel head: "Matching households" + count "47 households · 47 invoices · sorted by largest balance first". Households not students because reminders go to families, not individual students; one parent of two children gets one reminder, not two. The "47 households · 47 invoices" being equal here is coincidence — could be "47 households · 52 invoices" if some families have multiple children.
  • Bulk actions on the right: "Check all / Uncheck all / Sort: Balance ↓". Default is all-checked; admin uncheckes individuals. Uncheck all is in terra to signal it's significant — admin would only do that if they want to start fresh and check specific people. Sort dropdown gives ordering choice (balance, name, aging).
  • Excluded note bar in terra-soft above the table: "3 manually excluded · they will not receive this reminder · View excluded ↗". Sticky reminder of admin's manual interventions. Without this banner it'd be easy to forget which rows you'd unchecked while scrolling through the long list.
  • Recipient table columns: checkbox / Recipient / Match / Balance / Aged / Channels. Each column earns its space. Match column is the unusual one — shows the filter attributes that satisfied each row's match (class pill, status pill, aging pill). Admin can sanity-check the filter logic by scanning Match.
  • Match tags echo the filter-rail colour vocabulary. Class tag in neutral, Overdue tag in terra-tinted, Aging tag in warn-tinted (gold-amber for "this is the time-based one"). Visual continuity across the modal — admin sees the same colour grammar in the rail and in the result.
  • Header checkbox is in "partial" state (gold border, gold dash inside, no fill) because not all rows are checked. Tri-state checkbox — full (all on), partial (some on), empty (none on) — gives admin a quick read of the bulk state.
  • Unchecked rows are terra-tinted with reduced opacity (0.7). Same row layout, different visual state. Stays in the table rather than disappearing because admin might want to re-check after deciding "actually let me include Daniel." Hiding unchecked rows would force a different mental model and make restoration harder.
  • Channel cell shows two icons (WhatsApp + SMS) per row. When a recipient doesn't have WhatsApp opted in, the W icon is greyed; when SMS isn't reachable, the S is greyed. Patience Ofori's row shows a dim S meaning "her SMS number is iffy" — useful info before sending.
  • Excluded popover anchored to the "View excluded" link. Small floating card showing the 3 unchecked households with an italic "reason" line and a "Restore" button. The reason is admin-typed inline when she unchecks a row (a tiny inline input that appears on uncheck), preserved as audit trail.
  • Realistic exclusion reasons: "Reminded yesterday · already messaged" / "Spoke to him in person · paying Friday" / "Bereaved · holding off until next term." Real human reasons, not enum values. The bereavement case is operationally important — sending a fee reminder to a grieving family would damage trust irreparably.
  • Restore button per excluded row in italic gold uppercase. Quick reversal if admin changes her mind. Restoring removes the row from excluded and re-checks it in the main list.
  • Sticky modal footer with 3-cell summary + actions. "Will receive this reminder · 44 households" (italic gold "44") / "Sending cost · GHS 2.20" / Cancel + "Continue with 44 recipients →" primary CTA. Two stats + actions in a tight horizontal strip; everything admin needs to commit visible without scrolling.
  • "Continue with 44 recipients" CTA uses italic gold for the number. The 44 is the most important fact at this moment; treating it visually as the lever (italic gold) makes it pop. Same convention as user-tunable values everywhere.
  • "WhatsApp included in your bundle" sub-line on the cost stat. WhatsApp Business has conversation pricing where bundled conversations are effectively free; only SMS has marginal cost. Honest about why the cost is GHS 2.20 not GHS 22 (44 SMS × 0.05 = 2.20, no WhatsApp cost). Admin learns the cost model.
  • What's not on this surface: No save-as-segment ("save this as 'JHS 3 unpaid' for later"). No combine-segments ("union of segment A and segment B"). No A/B split ("send variant A to half, variant B to half"). No scheduled send ("send this Friday at 9 AM"). MVP1 keeps it simple; the modal builds one list, sends it once.
  • What's also not here: No bulk uncheck-by-attribute ("uncheck everyone in JHS 3A"). The use case is "remove individual outliers," not "remove a sub-group" — sub-groups belong in the filter rail, not in row-level interactions.
  • "Changes are not saved after this reminder is sent" lede line manages expectations. If admin spent 10 minutes constructing a list and unchecking individuals, she might assume the system would remember next time. It won't. Honest about ephemerality rather than letting admin discover the disappointment.
