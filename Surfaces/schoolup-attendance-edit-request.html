Approving the change

Most edit requests are simple — admin reads the dashboard line and clicks approve. But some need verification, sometimes there's an attached doctor's note, sometimes a correction SMS needs to go out. The deeper drawer is for the careful cases — diff before and after, the teacher's full reason, attachments inline, the consequences spelled out, an audit note for the record.

Why this layout

  • Two states show the spectrum of edit-request complexity. State A is the simple case — short reason, no attachment, no SMS impact. State B is the complex case — long reason, attached doctor's note, an SMS already went out, audit-worthy decision. Most real edit requests fall closer to State A; State B is the careful 1-in-5 that earns the deeper drawer.
  • Drawer head carries teacher identity, not student identity. The avatar is for the requesting teacher (Mr. Mensah, Ms. Adjei), not the affected student. This is fundamentally a request from one staff member to another — student is referenced but isn't the subject. Switches the mental frame from "fix this student's record" to "evaluate this colleague's request."
  • Teacher meta line shows their edit-request volume: "3 edit requests this term" / "1 edit request this term." Visible at the top so admin instinctively notices high-frequency requesters before reading the request itself. Mr. Mensah's 3 isn't unusual; Ms. Adjei's 1 is normal. The volume is context, not judgement — the system shows it without commentary.
  • Three-cell request meta strip in gold-tinted bg: Register · Submitted · Requested. Three timestamps that matter: which register's data is being changed, when the original mark was made, when the edit was requested. "52h late" / "72h late" in italic gold tells admin how stale the request is — was it caught quickly or has it been sitting?
  • The diff is the centrepiece. Two cards side by side: "Currently marked" (left, neutral bg) → arrow → "Proposed change" (right, coloured by target state — gold for Late, warn for Excused). Same colour vocabulary as the rest of attendance; admin instantly reads the change without parsing the words.
  • Status pill big in each diff card uses the saturated 38px square with letter (P, L, E, A) — consistent with the segmented control in the teacher marking flow. The visual progression: green → gold in State A, terra → warn in State B. The colour itself communicates the direction of the change before reading.
  • Detail line under the diff explains the original timestamp + reason: "Marked at 8:11 AM by Mr. Mensah · no reason logged" / "Marked at 8:18 AM by Ms. Adjei · no contact from home that day." Holds the original marking auditable; the after-state shows what's being proposed.
  • Student context card under the diff with avatar, name in display Fraunces, ID + class + guardian, and a colour-coded percentage tag on the right. Kwame at 94% (high green) is a normal student; Yaa at 78% (mid warn) is mid-range. Admin sees in one row how the change affects this specific student.
  • Reason block uses gold-bordered italic blockquote. Same pattern as the dashboard's edit-request preview. Admin already saw a snippet on the dashboard; the drawer shows the full text. Direct teacher voice — no editing or paraphrasing by the system. Attribution line below confirms who and when: "— Mr. Mensah · 9:18 AM, Friday 2 May."
  • Attachment card is the State B differentiator. Doctor's note rendered as a recognisable thumbnail — letterhead at the top ("KORLE BU TEACHING HOSPITAL"), printed lines, handwritten lines in terra ink, a circular stamp in the corner. The thumbnail looks like a real document, not just an icon — admin scanning the drawer immediately recognises that yes, there's a doctor's note here.
  • "⤢ open" overlay in the bottom corner in mono caps — clicks to expand. Two attachment-info buttons: "Open full size" (primary) and "Save copy". Admin can verify by inspecting the document; admin can keep a copy if the case is sensitive. Both flows respect that the attachment is a real piece of evidence, not just a UI element.
  • Attachment metadata is precise: "Uploaded by Ms. Adjei · photographed in person · photo.jpg · 2.4 MB · taken today 8:38 AM." Provenance matters — admin can see this was photographed today (not pulled from elsewhere), by the requesting teacher, of the original document. Reduces the chance of fabricated attachments.
  • Context grid: two cards, side by side. Student context (this student's recent attendance) and teacher context (this teacher's edit volume). Same canvas serves two questions — "Is this student's situation consistent with the request?" and "Is this teacher's request pattern normal?"
  • Student sparkline shows last 14 school days, colour-coded by status. Kwame's row is mostly green with two gold late-arrival bars; Yaa's row shows last week green then this week's 5-day excused warn streak. The sparkline corroborates or contradicts the request — Yaa's excused streak is consistent with Tuesday being a hospital day.
  • Teacher edit history bar shows 5 monthly tiles (Jan-May). Each tile's height = number of edit requests that month. Mr. Mensah has a small May spike (3 requests); Ms. Adjei has just one this whole term. Visualises whether the requester is a frequent editor without needing a number-heavy table. Anomaly detection without alarmism.
  • Consequence panel in gold-tinted card with icon. State A's icon is "i" (informational) — "No SMS impact." State B's icon is "!" (attention) — "An SMS already went out." Different icons signal the weight of the decision before admin reads the text.
  • Consequence text is operational, not legal. "The auto-absence SMS was sent to Cynthia at 3:00 PM Tuesday. Approving this change will reclassify the absence as Excused. Yaa's term attendance percentage stays at 78% (excused days count toward attendance)." Tells admin exactly what changes and what doesn't — no buried surprises after approval.
  • SMS-correction options as two tiles. "Send correction SMS" (selected, with example text) vs "No correction needed" (not selected, with reason). The example correction SMS is shown verbatim: "Yaa's Tuesday absence has been marked excused. Wishing her a speedy recovery." Brand-voice, warm, useful — admin can approve confidently knowing what the parent will receive.
  • "Wishing her a speedy recovery" is the kind of detail brands miss. Most systems would send: "Attendance correction: Yaa Amponsah's absence on Tue 29 Apr has been updated to Excused." Functional but cold. The system's brand voice extends to corrections — every parent contact is a relationship moment.
  • Audit note label shifts by stakes. State A: "Note for the record · optional." State B: "Note for the record · recommended for medical changes." The system gently nudges admin to write a note when stakes warrant it, without forcing. State B's note is pre-populated with a suggested format: "Verified Korle Bu doctor's note dated 30 Apr — paediatric ward discharge slip with hospital stamp visible. Admission and discharge dates match Ms. Adjei's account." Admin can edit or accept.
  • Audit-help line under the note tells admin where the note will live. State A: "Stored on the register's audit trail." State B: adds "also linked to Yaa's medical record on her student profile." Different consequences for different note types — medical notes propagate to the medical record; routine corrections only appear on the register's audit log.
  • Drawer footer carries the consequence summary on the left + actions on the right. State A: "Edit window closed · Tue 30 Apr · 8:11 AM" — admin sees the audit fact (the change is being made retroactively). State B: "Will trigger SMS · Cynthia · +233 24 555 0312" — the consequence is named with the actual phone number, so admin knows precisely whose phone the SMS will hit.
  • Three buttons in State B vs two in State A. State A: Reject + Approve. State B: "Request more info" (gold) + Reject + Approve. "Request more info" is a third path for cases where admin wants more from the teacher before deciding (e.g., the doctor's note is unclear, or the dates don't match). Sends back to teacher with admin's note as a question.
  • Approve button styling: green-bg, white text, ✓ icon, slightly fatter padding. Visual weight matches its consequence — approving is the affirmative action. Reject is white-with-terra-border, not solid red — rejection is a path, not a punishment. Both are equal-weight actions, but visually distinct.
  • State B's approve button: "Approve & send SMS ✓" instead of "Approve change ✓." The button label changes based on what it actually does — admin clicking knows they're approving the register change AND sending a correction SMS in one action, because the SMS option is part of the approve flow.
  • What's not here: No "compare with similar past requests" tool, no "ask another admin" path, no "request additional documents" workflow. Single-admin schools handle this themselves; multi-admin escalation is a future iteration. The drawer is built for the common case of "one person reviews, one decision made."
  • What's also not here: No risk scoring, no AI-suggested approve/reject, no fraud detection. The system surfaces facts; admin makes judgement. Trust is delegated to the human, not to an algorithm.
