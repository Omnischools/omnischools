Omnischools · Attendance · Alerts & follow-up

When the system speaks up

Attendance flags don't watch themselves. The system detects three patterns — long absences, low percentages, sudden shifts — and surfaces them across three places where admin already looks. Then the follow-up drawer turns the flag into a phone call, a WhatsApp message, a logged note, and a resolution. The dashboard ends with names; this is what happens after admin clicks one.

Why this layout

  • Three connected surfaces, one row. Detection rules feed the bell feed and the morning digest; same flag system, three places admin encounters it. Showing them side-by-side makes the data flow obvious — flags aren't a feature of any single surface, they're a system that surfaces in multiple places.
  • Detection rules visualised as readable thresholds, not toggles. Each rule shows the threshold ("3 days", "70%", "−30%") in Fraunces with the gold italic number, the consequence (Watching / Critical), and a plain-language description. Admin can see the whole logic at a glance — no wading through nested settings.
  • Three rule categories with iconographic markers — Long absence (terra L), Below threshold (warn B), Pattern shift (gold P). Same colour mapping as the dashboard's flag types, so admin already knows what they're looking at when they get here.
  • Watching → Critical escalation is two-tier. First trigger creates a "watching" flag (yellow). If the situation worsens, it escalates to "critical" (red). This matches how problems actually develop — gradually, not in single jumps. James's flag started watching on his second consecutive absence, escalated to critical when he hit five.
  • Bell dropdown shows the live activity stream. Three unread flags at the top with gold left-edge dots. Severity dots on avatars (terra critical, warn watching, green resolved) so admin can scan the stream and see proportions at a glance.
  • Each feed item has time, name, severity-coloured tag inline: "Escalated to Critical" / "New flag · Pattern shift" / "Marked Resolved by you." The verb tells the story; admin scanning the feed sees what changed, not just what is.
  • Resolved entries show in green for completion. Yaa Amponsah's flag was marked resolved — green dot, "Resolved by you" attribution, brief reason. Closure feels earned when the system shows it cleanly.
  • "Continuing to watch · pattern unchanged" entries (like Esi's) acknowledge that some flags stay open. Not every flag needs to close; some are slow-burn situations admin keeps an eye on. The feed reflects this without nagging.
  • Three tabs: Flags / Edits / All. Edits is the register edit-request stream from the admin dashboard — same notification system, different content type. "All" is the firehose for completists. Flags is the default because that's the most-actionable category.
  • Morning digest is the proactive surface. 7:30 AM email or WhatsApp message — admin sees it before reaching the office, before the dashboard, before students arrive. The digest is the "are there fires today?" briefing over coffee.
  • Digest paper styling mirrors the receipt and GES return. Navy crest, school name in Fraunces with gold italic, date in mono caps. Visual consistency across all the system's official-looking documents. Even though this is an email, it reads as authoritative.
  • Greeting addresses the reader by name and gives the headline number first. "Good morning Florence. Yesterday's attendance was 91.1% across 9 classes." Headline before details — admin who skims has the answer to "how was yesterday?" in the first line.
  • "Three things you may want to know about" frames the rest of the email. Conversational, not data-dumpy. The system is your assistant briefing you, not your dashboard yelling.
  • Yesterday's stat strip in the digest uses 3-cell layout with the same colour mapping (green attended, navy present count, terra absent count). Tighter than the dashboard's pulse strip because email screen real estate is limited; same semantic colours so admin recognises the language.
  • "New this morning" section uses terra-bordered cards with severity tags ("Escalated", "New"). Tags are tiny terra pills inside the name line — one-glance recognition that this is fresh, not last week's news.
  • "Escalated" vs "New" distinction matters. James was already on watching; he's now escalated to critical. Faith and Stephen are brand new flags. Different urgency, different mental response. The tag tells admin which bucket she's looking at without reading the body.
  • "Still being watched" section calls out flags with no recent follow-up activity. "Esi Asante · No follow-up logged this week." The system gently reminds admin of open items without being naggy. If admin acted yesterday, no reminder; if a week passed silently, a small prod.
  • "Open full dashboard" CTA at the bottom of the digest in navy with gold italic. The digest is the briefing; the dashboard is the workspace. Click-through is the natural next step when admin reaches her desk.
  • Footer explains why the digest exists. "Sent because you're the headteacher of Christ the King JHS · Adjust digest in Settings → Notifications." Compliance + opt-out path. Schools that don't want morning digests can turn them off; the digest itself tells you how.

Why this layout

  • Three states show the drawer's lifecycle in one view: fresh open (no history), logging (adding follow-up to existing thread), resolution (closing the loop). Reading left-to-right is a 3-day journey through how James's case actually unfolds.
  • Drawer header carries student identity + flag banner. Avatar, name in big Fraunces, class + ID + teacher. Below: a flag banner in the severity colour (terra for active, green for ready-to-resolve). The banner state changes as admin works through the flag — terra "no contact yet" becomes green "reached Mary" once a contact succeeds.
  • Banner text changes with progress. State A: "First flagged Wed 30 Apr. No contact from guardian." State B: "Flagged Wed 30 Apr · 2 follow-ups so far · still no resolution." State C: "Reached Mary · ready to resolve." The system narrates progress without admin re-reading the whole thread.
  • Recent attendance section uses the same expanded sparkline from the dashboard's attention list, but bigger. Four mini-stats above (Term / Last 14d / Streak / Last seen), then a 14-day grid where each day is a saturated colour block with letter inside. Same colour vocabulary as the rest of the system; admin doesn't relearn anything.
  • Day axis shows date stubs in mono ("M 21", "T 22", "W 23"…) so admin can correlate the pattern with calendar dates. James's streak is visible as five terra blocks ending at today; the Tuesday absence two weeks ago shows as another terra block. Pattern detection visible.
  • Three contact tiles: Phone, WhatsApp, SMS. Each shows the actual number/method, status, and an attribution. Phone with brand-styled handset icon, WhatsApp in WA-green with W mark, SMS in gold. Click any tile to launch into the existing flow (phone dial, open WhatsApp inbox, draft SMS).
  • Contact tiles show context-relevant detail: "opted in / Avg reply 14 min" for WhatsApp tells admin the channel works and is responsive. "Always reaches" for SMS reminds admin SMS is the lowest-tech but most-reliable channel. "+233 24 555 0421" on Phone is the direct number, ready to dial.
  • Empty history state in State A uses dashed-border card with explainer: "No follow-ups yet · This flag opened 2 days ago. Use the buttons above to contact Mary, then log what happened below." Tells admin what to do rather than just showing emptiness. The dashed border signals "this will fill up over time."
  • State B opens with Quick-Action grid — six chip buttons covering 95% of follow-up types: Called reached / Called no answer / Left voicemail / WhatsApp sent / SMS sent / Other. One click logs an action; the optional note field below lets admin add detail if there's any. The middle "Called · no answer" button is shown selected (gold-tinted) — admin clicked it to log her latest attempt.
  • Note input pre-populated with example text "Tried Mary's number — rang out twice. Will try again at 4 PM after school closes." Example shows the right level of detail — short, factual, includes next-step intent. Admin can edit or accept.
  • History timeline uses left-rail icons connected by a vertical line. Three entry types: system (grey "+" for "flag opened"), phone (navy with handset), WhatsApp (WA-green with W). Iconography matches the contact tiles so admin recognises actions across the surfaces.
  • Each history entry shows action + timestamp + note + author. "Called · no answer · Thu 1 May · 9:14 AM" with the note in italic gold-bordered block, then "— Florence Addo · Headmistress" attribution. Audit trail is human: who did what, when, with what result.
  • WhatsApp entry includes message status: "message delivered, not yet read." Pulls in real WhatsApp delivery state into the audit log. Admin who sent a WhatsApp Thursday afternoon and is now Friday morning sees that Mary still hasn't read it — informs the next action.
  • State C is the resolution state. Banner now green with checkmark icon. The system has detected (from State B's logged "Called · reached" action) that contact succeeded, and proactively flips the drawer into resolution mode. The drawer adapts to the situation; admin doesn't have to navigate to a different screen to close out.
  • Three resolution tiles: Resolved / Watching / Escalate. Watching selected (gold-tinted) because Mary said James will be back next Wednesday — not closed, but expected to close on a known date. Resolved is for "this is fully done"; Watching is for "I've handled this for now, let me check back"; Escalate is for "this needs district-level attention" (a rare path for genuine welfare concerns).
  • Check-in helper appears under the Watching tile — gold-tinted card with date input. "Check back on · Wed · 7 May 2026" auto-populated based on Mary's expected return date. Lightweight implementation of "schedule next check-in" as you specified — one date input, no calendar module.
  • Resolution note populated with structured summary: "Sick — doctor visit Tuesday. Mary will message us when James returns. Mark absences as excused for these days going forward." Captures the operational decision, not just narrative — the "mark as excused" instruction is a real policy choice that affects the term attendance percentage.
  • "What just happened" section shows only the most-recent contact (the call where Florence reached Mary). Truncated history because admin who's about to resolve doesn't need the whole thread visible — they need confirmation of the latest action that led to the resolution.
  • "What this resolution does" section explains the consequences in plain language. "Flag stays open in your dashboard with a 'checking back Wednesday' note. If James returns by then, the flag closes automatically; if he doesn't, you'll get a reminder Wednesday morning. Today's note is added to the audit trail." Contracts the future state so admin clicks Save with informed confidence.
  • Save button uses green-primary styling for resolution actions. Visual distinction from the navy "Submit" buttons used for ordinary commits — resolution is a positive state change worth celebrating.
  • Footer carries the resolution decision summary on the left. "Resolution · Watching · check Wed 7 May" so admin can verify before clicking Save. Two-line format matches the recipient summary on the send-reminders drawer.
