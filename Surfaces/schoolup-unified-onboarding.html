Setup wizard · unified · replaces 2 prior surfaces
The school onboarding flow · Basic stops at 6, SHS continues to 8

One onboarding.
One branch point. One layout.

The setup wizard is the school's first conversation with Omnischools — and until today it was two conversations, an MVP1 six-step "any school" flow and a later SHS-specific eight-step flow that drifted apart in chrome and step 4 content. This unified surface collapses them: one vertical step nav that scales to either tier, one shared step canon, one branch point at step 2 (school type) that activates the SHS-only steps. Step 4 adapts content based on the type decision — Basic gets classes + subjects, SHS gets programmes + cores + electives. Below: the wizard's full journey, in six demonstration states.

Why this layout

  • Vertical step nav, fixed width. Scales 6 → 8 → 10+ without redesign. Subtitles show what each step covers — no mystery boxes.
  • "Step 1 of 6 · or 8 for SHS" upfront. Honesty about scope: the SHS user sees they'll do 8; the Basic user sees they'll do 6. No bait-and-switch.
  • Steps 7–8 are visible but greyed. Branch transparency — anyone scanning the nav sees that more might come, with the dashed dot signalling "conditional on step 2".
  • CSSPS code field added. Single-field addition that lets SHS later flow placement lists through the admissions module from hm.cssps.gov.gh.
  • Required vs optional marked. Five of seven fields required at this step; the rest can be skipped now and filled later.
  • Auto-save behaviour. Filled fields tinted gold so the user knows the system is keeping them — typed value persists across Save & continue later.

The branch point

  • Five real options + one not-yet-available. TVET shown as dashed so the user knows it's planned without selecting it.
  • Step count visible per card. "6 steps" or "8 steps" right on the card — the user knows the cost of each choice before clicking.
  • Counter recalibrates on selection. Top of nav shifts from "Step 2 of 6" to "Step 2 of 8 · SHS path" the moment SHS is picked. Time-remaining estimate updates too.
  • Greyed steps brighten on next page. Steps 7–8 lose their dashed border once the user advances from step 2 — they become real steps in the plan.
  • Branch is one-way unless re-verified. Changing school type later is a Headmaster + GES code action, audit-logged. Prevents accidental tier shifts.

The same-step-different-content rule

  • Step 4 means "academic structure" universally. What gets entered differs; the step's purpose does not.
  • Step number stays consistent. Basic users at step 4 and SHS users at step 4 are doing the same conceptual work — the structural definition — at the same moment in onboarding.
  • Loaded defaults, not blank fields. Basic loads GES syllabus subjects per class; SHS loads WAEC programmes + cores + electives. The school confirms or edits, doesn't type from scratch.
  • Cores are universal. The four WASSCE cores (English, Math, Int Sci, Social) appear once at the top of the SHS panel; not duplicated under each programme.
  • Counts shown if available. The user can either pre-populate counts (if they know their cohort) or leave them for the admissions module to fill — both supported.
  • Step 4 sets up everything downstream. Programme rosters, the WASSCE module's subject matrix, gradebook structure, the timetable engine — all read from the choices made here. The wizard is honest about this in the help copy.

Basic completion design

  • Same wizard, different end-point. Achiase JHS completes at step 6; the SHS steps stay visible (greyed at 30% opacity) so the product story is honest — you can see what doesn't apply to you.
  • Celebrate without overselling. Green checkmark + "Welcome" tone. The school has done real work; the moment deserves acknowledgement. But no confetti, no fireworks — this is a tool, not a game.
  • "Go to dashboard" is the next step. Primary gold button, top-right of the celebrate card AND bottom-right of the page. The user shouldn't need to think.
  • Suggested next-steps grid. Four cards in priority order: admit students, invite parents, build timetable, customise settings. The school doesn't have to figure out what to do next.
  • Setup time shown. "11 min · 12 estimated" — honest accounting. Builds trust in time estimates for the next admin who will set up another instance.

Step 7 design

  • Three residency models · honest percentages. Day 7%, mixed 81%, boarding-only 12% across Ghanaian SHS. The default option ("Mixed") is highlighted because it matches reality for 8 in 10 schools.
  • House system is optional but shown by default. Not every SHS uses houses, but most do. The four-house pattern (named after school patrons) is pre-populated for the school to confirm or rename.
  • Module activation flagged in the help copy. The choice here drives whether Boarding, Sickbay, prep-attendance, and discipline-by-residency modules turn on. The user sees this consequence upfront.
  • Visiting day is a setup decision, not a per-event one. The cadence locks here; individual dates flow from the calendar module.

Step 8 design

  • WAEC centre code is the unlock. Without it, the WASSCE module stays hidden. The school provides it once.
  • Programmes inherit from step 4. No re-typing — the cards reflect step 4's choices, marked ACTIVE.
  • Calendar sync defaults to auto. Omnischools ops curates the WASSCE master timetable annually from WAEC's PDF; the school doesn't maintain it. Manual override available for centre-specific anomalies.
  • Regional office is significant. Drives where SC-12 make-up sittings happen — the school's accompaniment pattern depends on knowing this in advance.
  • First cohort year visible. A school onboarding in 2026 with current F1 students will see WASSCE for them in 2028; the module knows when to activate.

Why a final review

  • One screen, all decisions. Eight cards summarising eight steps. The Headmaster can confirm what's set without scrolling back through the wizard.
  • Edit-from-here. Every card has an "Edit →" affordance; clicking it jumps to that step. No multi-step "back" navigation.
  • Launch is a deliberate action. Gold primary button, top-right and bottom-right of the page. Until clicked, nothing in the school's instance is "live".
  • Post-launch settings location stated. "Find these under Settings → School profile" sets the expectation that changes are possible but happen elsewhere.
  • Re-verification flagged. Changing school type requires Headmaster + GES code — the most consequential setting is the hardest to change, the rest are just "Edit".
