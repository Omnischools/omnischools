Why both views matter · same students, different lenses

A student belongs to both structures · perpendicular by design

Academic axis · class

Form 2 General Arts A

What happens here: lessons · subject teaching · timetabling · grades · WASSCE preparation · academic reports · parent-teacher meetings for academic performance.

Run by: Form Master + 9 subject teachers · Mr Yaw Boateng (CRS) is Form Master.

Pastoral axis · House

The roster surface above shows the academic axis (32 students who are all in Form 2 Arts A) with the pastoral axis visible per student (which House each belongs to). The House view — accessed by clicking "House view" in the top bar — flips the lens: shows all 200 students of Knight House across all Forms and programmes, with their class affiliation visible. Same data, different organising principle.
For the build phase

The roster surface uses a single SQL query joining students × classes × houses × house_assignments through indexed foreign keys. Filtering by class is fast (~30 students). The reverse query — "all students in Knight House" — joins the same tables differently and might return 200 rows; Postgres handles both query shapes well as long as the right composite indexes exist. One database, two views, both efficient.

Calls worth flagging

  • The "unique combination" tag in the class identity strip is doing real work — admins occasionally try to create duplicate classes ("Form 2 Arts A" twice) and the system catches this via the (school_id, form, programme_id, section) composite unique constraint. Visible reassurance that the schema enforces uniqueness.
  • The "all 6 represented" House mix indicator is a subtle health check — if a class only has students from 2 Houses, it suggests the admission auto-balancer is failing. Surfacing the diversity nudges admin to investigate. Not an error, just a soft signal.
  • House dots in the table are small (11px), by design — they're visual markers, not the primary identifier. The House name to the right of the dot is what reads. The dot is for fast pattern scanning when the admin's eye is moving down the list.
  • The quick actions card in the right rail is gold-tinted because it's the "what next?" hub — Form PTA scheduling, report card generation, grade analytics. Most admin work after viewing a roster involves one of these.
  • "View timetable" and "House view" sit in the class-identity strip as peer affordances to the roster — they're three lenses on the same class entity (academic schedule, student roster, pastoral makeup). Lateral navigation, not hierarchical.
