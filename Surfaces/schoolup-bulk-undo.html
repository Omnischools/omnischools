Undo is not a magic word

The right answer to "can I undo this?" depends on what was done. Bulk class transfers — yes, one click. Bulk SMS sends — no, the message is gone. Bulk receipts issued — no, the financial record is append-only. The honest design tells admin which is which before they commit, then provides one-click undo for the reversible category and clear correction paths for the rest. No false promises, no buried disclaimers.

Editorial principle

Honesty about reversibility is more valuable than universal-undo-that-doesn't-actually-work. Admins build trust in the system when "can I undo this?" has a knowable answer at commit time, not a hopeful answer they discover later.

Calls worth flagging

  • The reversibility tier shows up in the bulk-operations confirm dialog, before commit — so admin sees the green/warn/terra badge alongside the preview. Surface the table once, link to it everywhere it matters.
  • Tier B is the editorial fork: "undone if uncommunicated" is a specific contract. The button greys out the second Omnischools records the cascade event (SMS sent, parent notification fired).
  • Tier C correction paths are not failures — they're the right answer. Void receipts are how bookkeeping has worked for centuries. Omnischools implements the centuries-old answer cleanly.
  • Correction SMS prefix "Correction:" is not enforced — admin can edit. But it pre-populates because clarity to parents matters more than admin's freedom to phrase.
  • Bulk transcript reissue stamp follows the same model as void/refund receipts — original reference preserved, replacement explicit. Same pattern across the system.

The pattern, mirrored

The bulk-operations file established the four-step pattern: selection → confirm → result → audit. Undo follows the same four steps in reverse — admin sees a preview, commits with reasoning, gets a result card, and the action lands in the audit log alongside the original. The system stays predictable.

Calls worth flagging

  • "Reason for undo" is optional but recommended — same pattern as voids and refunds. Florence isn't required to explain herself, but the field is there because reasoning is the most valuable part of an audit trail.
  • The placeholder text in the reason field is a concrete example ("Promoted wrong cohort — should have been..."), not lorem ipsum. Admin sees what a good reason looks like.
  • "Undo cannot itself be undone" is small text in the modal foot — once you click Undo, you can't un-undo. To revert again, you'd have to redo the original bulk action manually. This is intentional friction.
  • Result card shows three stats, not one: records restored, side-effects rolled back, comms triggered (em-dash because nothing fired). Demonstrates the rollback was complete.
  • The audit log link is the closing loop. Florence can verify the undo is recorded, not just promised.
04
When undo isn't possible · the correction paths
Tier C operations · forward-only correction
SMS

Bulk SMS / WhatsApp · send correction

The message has reached the MNO. There is no recall.
Scenario: Florence sent a fee reminder to 312 parents at 16:04 yesterday. She just realised the amount in the message was wrong (GHS 340 not GHS 380).
The correction path:
  1. Click "Send correction" on the original op in Recent.
  2. Omnischools opens compose with recipient list pre-populated (same 312 parents).
  3. Message is pre-prefixed with "Correction:" — Florence writes the new content below.
  4. Send. Both messages stay on record · parents see two messages, the second supersedes the first.
RC

Bulk receipts · issue void

Receipts consume sequential numbers · GRA needs the chain unbroken.
Scenario: Joyce ran bulk receipt issuance for 47 students three days ago, but five of those payments turned out to be duplicates that had been recorded twice.
The correction path:
  1. Open each affected receipt individually.
  2. Click "Void this receipt" — issues a new void record referencing the original.
  3. If money was actually moved, issue a refund via the same drawer (next file in the existing void/refund flow).
  4. Original receipt stays · void receipt links to it · GRA-ready chain intact.
TR

Bulk transcripts · reissue with stamp

University registrars rely on the latest issued transcript.
Scenario: Florence batched 28 graduating-cohort transcripts last term. Three students had grade corrections after the batch was already printed and signed.
The correction path:
  1. Open each affected transcript record.
  2. Click "Reissue" — generates new transcript with corrected grades.
  3. New transcript carries footer: "Replaces transcript GHA-2024-0142, issued 12 Apr 2024."
  4. Original number stays on record. Reissue carries new number with cross-reference · registrar always uses the latest.
AD

Bulk admission decisions · manual reversal

Once parents are told, the message is in the world.
Scenario: Florence accepted 18 applications and SMS confirmations went out within minutes. She then realised the cohort was over-filled — three of those acceptances need to be withdrawn.
The correction path:
  1. Open each affected application individually.
  2. Click "Reverse decision" — opens compose drawer for parent comms (this is not optional; parents must be told).
  3. Omnischools pre-populates a "We sincerely regret..." template · admin edits and personalises.
  4. Application status returns to "Pending" · audit log records both the original decision and the reversal with reasoning.
Why this matters

Every correction path follows the same pattern: the original event stays, a new event references it, the new event is forward-only. This is how banking, accounting, and medical records have worked for decades — Omnischools implements the established answer rather than inventing a new one. Auditors, GRA, and university registrars all expect this shape.

Calls worth flagging

  • The four cards cover the four Tier C operations from Section 01. Each scenario is concrete, named, and grounded in a plausible mistake — not abstract "what if" prompts.
  • SMS correction prefix "Correction:" is editable but pre-populated. Different vocabularies might suit different schools (a private school might prefer "Update:") — the prefill is the recommended default.
  • Void receipt + refund flow already exists in MVP1 (see schoolup-void-refund.html). This file just shows where bulk leads to it.
  • Transcript reissue footer text is not customisable — the wording matters for university registrars and should be consistent across the platform. One vocabulary, learned once.
  • "Reverse decision" for admissions requires parent comms — there's no silent reversal path. Omnischools enforces this because reversing an admission silently is a worse outcome than reversing it visibly.
