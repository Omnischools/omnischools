Omnischools · Inbox

When parents reply

WhatsApp Business is two-way. Parents reply to template messages, ask questions, share concerns. The inbox is where school admin sees those replies and answers them. The 24-hour conversation window mechanic shapes everything: when it's open, you can reply with anything; when it closes, only approved templates can re-engage.

Why this layout

  • Inbox added as top-level sidebar nav with an unread count badge (7). The inbox is the response surface for an entire two-way channel; nesting it inside Announcements would underplay it.
  • Three-pane layout: list / conversation / context. Email-style two-pane wouldn't be enough — answering a parent question well needs the student's billing status and grades visible without leaving the conversation. The right pane is the school context that turns an okay reply into a great one.
  • Filter tabs: Urgent / Open / All / Unread / Closed / Mine. "Urgent" is first and styled in terra — these are conversations with closing windows or unread messages, the highest priority. "Mine" prepares for MVP2+ multi-admin assignment.
  • Conversation rows show window status three ways: small status dot on the avatar (green/gold/grey), window pill in the bottom-right (Open / 22h left / Closed), and the row treatment. Multiple visual reinforcements because the window state is consequential.
  • Student context line under the name: "re: Akosua Boateng · JHS 2A · payment dispute" combines the student's name with a topic. Prevents the "wait, who's Patricia?" moment when scanning.
  • Conversation pane uses WhatsApp visual conventions: chat-background colour with subtle pattern, day dividers in warm yellow, school's outbound in green-tinted bubbles, parent's inbound in white. Familiar mental model for admin who use WhatsApp daily.
  • Window strip immediately below header — green when open, gold when closing soon, grey when closed. Always-visible reminder of whether free-form reply is allowed. "22h 18m left" is precise so admin knows whether to type now or send a template.
  • Template tag on outbound template messages ("Template · payment_confirmation_v3") in green caps inside the bubble. Distinguishes templated messages from free-form replies — useful in audit and in spotting where a conversation began.
  • Internal notes inline as gold-bordered blocks. Critical for audit. Admin called Patricia, made notes — that activity becomes visible in the conversation flow without going to the parent. The 🔒 lock icon and "not sent to parent" text removes ambiguity.
  • Compose-mode toggle: Reply (green) vs Note (gold). Mode-aware visual treatment carries through: in note mode, the input field tints gold and the send button becomes gold. The mode is an explicit choice; admin can't accidentally send an internal note as a reply or vice-versa.
  • Quick-reply chips above the input. "Acknowledged", "Will look into this", and a context-aware date chip ("Mock exams: 19-23 May" — auto-generated from a prior internal note). Click to insert; admin can still edit before sending.
  • Right context pane: Guardian card, Children at this school, Quick actions, Conversation metadata. Each card answers a different question. The fee-status pills tell admin Akosua is paid up but Kofi is at 60%; quick actions let admin jump to billing/grades without losing the conversation.
  • Conversation cost shown in the context pane: "Cost so far GHS 0.18" — single conversation cost regardless of message count. Helps admin internalise WhatsApp's economics.
  • "Assign to colleague" button is present but flagged for MVP2+. The wireframe shows where it lives so the data model supports it from day one, even if multi-admin assignment doesn't ship in MVP1.

Why this layout

  • Closed window has dramatically different compose treatment. Instead of a text input that admin might mistakenly use, the entire compose area replaces with a closed-banner + template picker + SMS fallback. The constraint is unmissable.
  • Window strip in grey shows the closed state with precise duration ("2 days, 4 hours ago"). Same component as open-state; the visual treatment shifts so admin doesn't have to read words to know the state changed.
  • Closed-window banner explains the constraint in plain language. "Free-form replies aren't available... once he replies, the window opens for 24 hours and you can reply with anything." Teaches the WhatsApp model without lecturing.
  • Suggested templates ranked by context. The previous conversation was a PTA meeting reminder; the suggested templates start with general_followup (most reusable), then templates that fit similar contexts. Reduces decision time when admin knows what to send.
  • SMS fallback is the escape hatch. Sometimes WhatsApp templates don't fit and the message is urgent. The SMS fallback option in gold makes the trade-off explicit: "Costs ~GHS 0.05 and reaches him immediately without a window mechanic." Honest about cost and character limits so admin chooses informed.
  • Window timeline in the context pane shows the lifecycle — when the window opened (and by which template), when the parent last replied, when it closed, how long ago. Auditor reconstructs what happened in one glance.
  • Same conversation history is still readable above. The closed state doesn't hide history; admin can still scroll through past messages. Only the forward action (composing) is constrained.
  • "Status: Closed" annotation in the cost card. The cost is GHS 0.18 because that's what the conversation cost when it was active; the conversation isn't accruing more cost while closed. Admin sees the bill is settled.
