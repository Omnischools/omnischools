Omnischools · Void & refund flow

Correcting payments safely

Mistakes happen. Parents dispute. Students withdraw. Omnischools lets admins void and refund payments — but always with reason text, audit history, and a replacement receipt. The financial record is corrected by overlay, never by silent edit. The financial audit log lives under Billing for the accountant; a separate activity log lives under Settings for admin oversight.

Void

The record was wrong

A payment was recorded incorrectly — wrong amount, wrong student, duplicate entry. The original receipt is voided; if the money was correctly received, it's typically re-recorded against the right invoice.
  • Admin typed GHS 4400.00 instead of GHS 440.00 — void and re-record
  • Payment was recorded against Yaw's invoice, but the parent paid for Akosua — void and re-record correctly
  • Same MoMo transaction was recorded twice by mistake — void the duplicate
Refund

The school is returning money

The original payment was correct, but the school is sending the money back to the parent. Original receipt stays valid; a refund event is recorded separately, and the invoice balance reverts.
  • Parent disputed a MoMo charge and the school agrees — refund via MTN reversal
  • Student withdrew mid-term; school refunds the prorated balance
  • Parent paid GHS 500.00 against a GHS 400.00 balance and asks for the GHS 100.00 back rather than leaving as credit

Why this layout

  • Two-column layout separates the static facts from the running history. Left column is "what this payment is"; right column is "what's happened to it."
  • Three actions in the page header. "Download receipt PDF" is the most-used; "Void or refund..." is the danger action with terracotta outline. Trailing ellipsis signals "this opens a flow, not a one-click."
  • Status pill in the card header shows Confirmed (green dot). When voided, the same card would show "Voided" (terracotta dot) — same surface, different state.
  • "Applied to" allocation list matches the multi-line model. For a single-invoice payment it's one row; for multi-invoice it would show all targets. Same component, both cases.
  • Receipt mini-card with three actions (View, Download, Email) lives in its own panel because the receipt is the document that goes outside Omnischools. Distinct from the payment record itself.
  • Audit timeline reads top-to-bottom most-recent-first. The action that just happened is at the top — the one admin needs to see most. Created event sits at the bottom as the foundation.
  • Each audit event has icon + what + when + who + optional detail block. Five fields, each load-bearing. The detail block is grey for ordinary events, terracotta for voids/refunds.
  • Subtle events still show up ("Receipt viewed by guardian"). For a school product, knowing the parent actually saw the receipt is genuinely useful — proves notification worked.

Why this layout

  • Title is "Void or refund?" with a question mark. The drawer's first job is to help the admin distinguish the two. The body of the drawer is structured around answering that question.
  • Refund tile selected by default in this state. Showing the refund branch because it's the more common case (parent disputes, student withdrawals are weekly events; data-entry errors are weekly too but tend to be void). In production, no default — admin actively picks.
  • Void tile uses terracotta; refund tile uses gold. Distinct visual signals because they're conceptually different: void is "this was wrong" (red), refund is "we're sending money back" (warning gold).
  • Tile descriptions explain consequences in plain language. Not "the receipt is invalidated" — instead "the receipt stays valid. A refund record is created and the invoice balance is restored." Admin learns the model from the description.
  • Refund amount field with default = full payment. Most refunds are full; partial refunds (mid-term withdrawals) need to override. The hint text explains.
  • Reason is required, with preset chips. Free-form text is the canonical record; presets are quick-fills for common cases. Admin can tap "Parent disputed charge" then add specific detail. The reason text appears on the void/refund receipt itself — this is what the audit reads.
  • Method field captures how the money actually moved. "MTN MoMo reversal · TXN-REV-78215" lets the school trace the refund back to MTN's records. Cash, bank, cheque all work too.
  • Consequences strip in terracotta-bordered block reads back exactly what's about to happen, in four bullets. Same pre-flight pattern as the record-payment drawer; same level of trust before a destructive action.
  • Confirm checkbox at the bottom requires explicit acknowledgment that the action is durable. Adds a small piece of friction that earns its place — refunds shouldn't be undoable with a back arrow.
  • "Confirm refund of GHS 340.00" on the gold danger button — anchors the action to the specific amount, prevents wrong-amount errors at the moment of commitment.

Why this layout

  • Lives under Billing, not Settings. The financial audit is an accountant's daily/monthly tool — putting it where the money lives matches the workflow. Accountants navigate Billing → Financial audit, not three layers deep into Settings.
  • Period selector at the top defaults to current month. End-of-month reconciliation is the most common reason to open this page; the default does the right thing without a click.
  • Financial summary strip shows the period totals at a glance: total received, broken down by method, plus voids/refunds in terracotta. This is what an accountant copies into their reconciliation report.
  • "By MoMo / By cash / By bank transfer" breakdown matches how schools actually account — the accountant reconciles each channel against its source (MTN settlement statement, cash drawer count, bank statement). The sum of these tiles plus the refund delta equals net cash position change.
  • "Every cedi in & out" page title sets scope. Plain language; tells the accountant exactly what they'll find here. Voids, refunds, and discounts are visible alongside payments because they're all monetary events.
  • Discounts in the financial audit, not the activity log. A discount changes what's billed — accountants need to see them when reconciling expected vs received. Even though they're not cash transactions, they're financial events.
  • Voids and refunds tinted soft red. Same row treatment as before; subtle but unmistakable when scanning.
  • Sub-line carries forensic detail. Each "what happened" row's second line includes the receipt number, MoMo TXN, bank reference — whatever the auditor needs to cross-check against external records.
  • Two export buttons: CSV (for spreadsheet reconciliation) and PDF (for filing/Ministry submission). Same data, two formats.

Why this layout

  • Page title "Who did what, when" is the auditor's question, plainly stated. Activity log is fundamentally about accountability — who took which action, on what subject. The title matches.
  • Cross-link to Financial audit log at the top of the page. An admin who lands here looking for a payment record sees immediately where to go. Reduces wasted time and prevents the wrong column being scanned.
  • No amount column. Activity events don't have monetary values; forcing an amount column would create empty cells everywhere. Five columns instead of six: When / Event / What / Subject / By.
  • "Subject" column instead of "Student" because activity touches more than students. Settings changes have "School-wide" in italic grey; role changes have the new staff member as subject; teacher invites have the teacher as both subject and actor (their first action was accepting their own invite).
  • Four event categories with their own colors: Endorsements (gold), Student edits (green), Roles & access (navy), Settings (navy). Different palette from the financial audit's red/green/gold — visual distinction reinforces the conceptual split.
  • Activity log is admin-only. Accountant doesn't see this nav item — they have no use for it and surfacing settings changes to a billing-only role creates noise. Reflects the same role-permission model from the accountant role wireframe.
  • Student edits include both data updates and class moves. "Guardian phone number updated" and "Class assignment changed" are both consequential changes to a student record. The audit captures both with the same tag.
  • Access events ("Mr. Mensah accepted teacher invite", "2FA enabled") matter for security audits. If a school suspects unauthorised access, this is where they look first.
  • Same audit principles apply. Append-only, no edit/delete on rows, exportable CSV, the original action gets a corresponding correction entry rather than a silent edit.
