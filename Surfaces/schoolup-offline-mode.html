Omnischools · Deferred surface
MVP 2 · post-launch

Connectivity is a question, not an answer

Rural Ghanaian schools are at the end of patchy 3G networks; power cuts kill routers; metered mobile data makes admin think twice before opening anything. Omnischools's offline mode isn't a "fallback" — it's the design starting point for the operations that have to work when the network doesn't. Attendance is the flagship case. Marking 312 students absent because the network dropped is not a feature.

01
What works offline · what doesn't
10 operations · 3 capability tiers
The principle

Some operations are local first. Others can't be.

Omnischools installs as a Progressive Web App — capable of caching the school's roster, attendance form, and gradebook locally. Operations that only need the school's own data (mark attendance, save grades, draft a parent message) work offline against the cache. Operations that need the outside world (send SMS via MNO, process MoMo payment, fetch a new student transfer record) wait for connectivity.

The capability map below tells admin which is which, before they need to know.

Why these tiers, not "online vs offline"

A binary "the app is offline" banner is wrong because it suggests nothing works — when in fact most of the day's work (taking attendance, saving grades, looking up Akosua) works fine. Tier-aware design lets the system grey out only what truly can't work, while keeping everything else silently functional. The teacher takes attendance during the morning power cut; she only notices something different when she tries to send a message.

Calls worth flagging

  • 14 days of queued attendance covers worst-case scenarios — a school with no power for two weeks (rare but real). Beyond 14 days, admin sees a warning to reconnect or risk data loss.
  • Cache size ~28 MB total is calibrated for cheap Android devices with limited storage. Larger schools (1000+ students) auto-tune the cache to keep only the active term's gradebook.
  • Cash payments queue with a locally-reserved receipt number from a pre-allocated pool. When the queue syncs, the numbers are committed in order. No duplicate receipt numbers — the chain stays intact for GRA.
  • MoMo payments cannot queue — pretending they could would create false expectations. The honest answer is "MoMo needs network; tell the parent to retry or accept cash."
  • This file describes what offline mode does. The product decision whether it's an MVP1 or MVP2 feature remains open — for a flagship attendance system in rural Ghana, MVP1 may be the right answer.

Why the pill, not the banner

A full-width red banner saying "YOU ARE OFFLINE" is what most apps do. It's wrong for Omnischools because it implies the app is broken — but the teacher is fine, attendance is being taken, grades are being saved. The pill is honest without being alarmist: connection is one part of the system; here is its current state. Calm signal beats panic banner, especially in environments where the offline state is the default state.

Calls worth flagging

  • Pill placement: top right of the nav bar, near the user avatar. Not in the breadcrumb, not as a banner. Calm location for calm signal.
  • The pulse animation on the syncing dot is 1.5s opacity cycle — slow enough to be peripheral, fast enough to read as "active". Faster pulses read as alarms.
  • The terra "needs your attention" state explicitly does not use the word "error" in the pill itself — "2 changes need review" is a task, not a failure. Helps admin act without panic.
  • Connection state is checked every 30 seconds when active, every 5 minutes when idle. Mobile data isn't free — we don't poll aggressively.
  • Sync queue persists across browser refreshes. Admin can close the tab, the queue waits, sync resumes when they reopen on connection.

The queue is the trust artefact

Most of the time admin doesn't need to look at the sync queue — the system just works. But on the rare day when something doesn't sync, the queue gives admin a complete picture: what was captured, when, by whom, what happened. No black boxes. If two changes fail, the rest of the day's work is still uploaded — partial sync is normal, not a failure.

Calls worth flagging

  • The queue shows changes in capture order, not sync order. Helps admin reason about "what was happening this morning" — not "what got lucky with the network first".
  • Cash payment queued at 07:55 with receipt number RC-2025-0847 — that number is committed locally and won't be reused. Receipt chain stays intact even when admin records cash during an outage.
  • SMS sync row shows "3 of 47 dispatched" — the MNO accepts in batches. Admin can see real-time progress through the bulk send.
  • Conflict rows are terra-tinted in their entirety, not just the badge. The whole row signals "this one needs your attention" — clicking opens the conflict resolution UI (Section 04).
  • Successful uploads stay in the queue for 24 hours before being trimmed — long enough that admin can audit "did everything sync?", short enough that the queue stays scannable.

Why admin chooses · not the system

The technically easy path is last-write-wins — newest edit silently overwrites the older one. Omnischools doesn't do that for records that matter (attendance, grades, financial lines) because "newest" is not the same as "right". Mrs Akua's later edit is probably right (she has the note about the bus). Mr Yaw's later edit is probably wrong (he didn't know about the participation adjustment). The system's job is to surface the conflict honestly; the school decides what's true.

Calls worth flagging

  • The recommended version (gold border) uses simple heuristics: more recent, has explanatory note, came from the device closer to the source of truth (form teacher for attendance, subject teacher for grades).
  • The "Decide together" button on the grade conflict opens a shared resolution thread — Mrs Abena and Mr Yaw can chat about which score is right before either one is committed. Useful for non-trivial conflicts.
  • Omnischools does not auto-merge for records with audit weight. Free-text notes can be merged (concatenate); structured fields with conflicts need a human decision.
  • The conflict UI is reachable only from the queue inspector — not pushed as a modal interruption. Admin reviews when ready, not when the system decides.
  • Once resolved, the audit log records both versions and the decision: "Conflict resolved: Mrs Akua's Late kept, Mr Ebo's Present discarded by Florence A. on 2 May 08:42 — reason: bus delay confirmed." Forward-only audit trail; no edit erased.
