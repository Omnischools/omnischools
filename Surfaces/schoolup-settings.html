Why this layout

  • Two-level architecture: a directory landing page (this state) and individual deep pages for each section (next state + the attendance settings already designed). The landing page is a wayfinder, not a kitchen sink. Most admins land here, scan, click into one section, do their thing, leave.
  • "School settings" headline with personalised "Christ the King JHS" in the lede line. The lede sets honest expectations: "some sections need attention; most are set and forgotten." Settings is mostly maintenance; this isn't a daily-driver surface.
  • Health strip with 4 tiles at the top. Dense at-a-glance summary: School info complete · 2 need attention (stamp + roles) · 3 connected services (Hubtel/WhatsApp/SMS) · 8 active users. Admin sees the configuration health in one glance; she clicks into "2 need attention" if she wants to triage.
  • Health-tile colour vocabulary uses the locked palette — green-bg + green icon for complete, warn-bg + warn icon for problems. Same convention as everywhere else in the system; admin knows what each colour means without instruction.
  • 5 grouped sections numbered 01—05 with display Fraunces titles. The school / Daily operations / How the school speaks / Access & security / External integrations. Group order roughly follows frequency of access — school basics rarely change, daily operations change each term, communications and access mid-frequency, integrations rarely.
  • Group titles are evocative not bureaucratic. "How the school speaks" not "Communications settings"; "Daily operations" not "Operational configuration." Settings UI doesn't have to be sterile — display Fraunces with italic gold makes the group titles feel intentional rather than autogenerated.
  • Group-meta in tracked-out caps ("Set once · rarely changes" / "Touched each term · per cadence" / "Voice · channels · privacy" / "Who can do what" / "Third parties & data sharing"). Tells admin how often each group needs attention; she can mentally bucket "do this now" vs "set up and forget."
  • Each card: icon + name + description + state row. Consistent layout means admin learns the pattern once. The state row at the bottom shows current configuration status (complete / warn / specific count) on the left and "last edited" on the right. Last-edited dates anchor the cards in time — admin sees what's recent vs stale.
  • Card icons in coloured tiles. Navy/gold for school/access/login (institutional weight), gold-bg for branding/structure (configuration), green-bg for billing (financial), terra-bg for roles (security boundary), Hubtel-blue for the Hubtel card. Functional colour-coding rather than purely decorative.
  • Branding card has terra-shifted border + warn dot in top-right corner because stamp is missing. The visual signal precedes the text — admin scanning the grid sees the warn dot and the terra border, knows where to focus, then reads "Stamp missing" in the state row.
  • Roles & access card in the same warn state because there's an inactive user (Mr. Yeboah, who left in May, still has an account). Stale accounts are a real security issue — the warn state surfaces the housekeeping rather than letting it accumulate silently.
  • State row text mixes plain English and specific values: "Complete" / "14 templates · 11 active" / "Mr. Yeboah · left May" / ""CHRIST KING" · 2 channels live". Each state line tells admin what's set without making her open the page to find out. She can decide "do I need to drill in?" from the card alone.
  • "3 connected" health tile shows Hubtel + WhatsApp + SMS as the live integrations. The integration cards in Group 05 echo this with their own state. Connected services have authority on the page — they're the lifelines that keep operations running.
  • Group 05 shows two cards (Hubtel + GES data sharing) in two-up grid, not three-up like other groups. Card density adapts to content; integrations card is a different shape than configuration cards because the meta data is denser.
  • "GES & ministry data sharing" card is genuinely about MoE pitch context — schools sharing aggregate stats for unified performance monitoring (the original product vision). The card sits in Settings because the school controls what's shared. "Anonymised by default" in the description sets the privacy posture before admin opens the deeper page.
  • Settings landing has no save bar because nothing is being edited here — it's a directory. The save bar appears on the deeper pages where editing happens. Visual rhythm matters: directory → no save bar; configuration → save bar at bottom.
  • What's not on the landing: No global search ("find a setting") — 10 cards is scannable enough. No favourites or pins (admins don't have favourite settings). No "recently changed" timeline (last-edited dates per card serve that purpose).
  • What's also not here: No system status, no usage analytics, no billing for Omnischools itself. This is school-configuration; Omnischools account billing would live elsewhere if it existed at all (likely as a separate page accessible from the user menu, not Settings).

Why this layout

  • Same head architecture as attendance settings: crumb + headline + lede + tabs + body. Admin who's been to attendance settings already knows this layout. Settings is a family of pages, not bespoke designs per section.
  • Eight tabs across the top — School info / Branding / Billing / Attendance / Notifications / Roles & access / SMS & WhatsApp / Integrations. The tabs are the same set everywhere in Settings; clicking any switches the whole content area but keeps the tab strip stable.
  • Three regions stacked vertically: School identity / Logo & stamp / Academic structure. Frequency of touch order: identity changes are rare (a school renaming is a major event), branding can change occasionally (new logo design), academic structure changes once a year (new term dates). Most-stable to most-changing.
  • Region 1 — School identity uses 8 field cards in a 2×4 grid. Each card: label / value / meta / edit link. The values are Fraunces display for proper-noun fields (school name, type) and JetBrains Mono for identifier fields (GES ID, phone, email). Visual distinction between human and machine values.
  • "Short name · for SMS & UI" field with truncation note: "Truncated to 'CHRIST KING' in SMS sender ID." Connects this setting to its consequence elsewhere — admin sees that the short name is what parents recognise in their SMS inboxes. The locked truncation rule we established earlier surfaces here.
  • "GES School ID · cannot be edited after registration" field has no Edit link. Some fields are immutable by design — the GES ID is assigned by the government and changing it in Omnischools would create chaos. The card visually signals immutability by lacking the Edit affordance.
  • Region 2 — Logo & stamp in side-by-side cards. Each card has the same structure: eyebrow + headline + description + visual preview + file details + format guidance + usage list. The stamp card has "attention" in its eyebrow reflecting its warn state from the landing page.
  • Logo preview shows the actual logo treatment — navy/gold "CK" tile + Christ the King JHS in display Fraunces with italic gold "JHS" + "EST. 1962 · ACCRA" tagline. The preview IS the logo system; admin sees how the logo will render across surfaces.
  • Stamp preview shows the actual generated stamp — terra circular border at -6°, 3-line content (Christ the King / Junior High School / ACCRA · 1962). Generated from school name when no stamp is uploaded; the placeholder is functional, not decorative. Admin understands she can use this if she doesn't have a digital scan.
  • "Photograph or scan the school's official rubber stamp on plain white paper" guidance tells admin exactly how to digitise. Most schools don't have a designed digital stamp; they have a physical rubber stamp. The guidance bridges the analog-to-digital gap honestly.
  • Format guidance has 3-4 specific bullets per asset. PNG with transparency / minimum dimensions / what works visually / what to avoid. The guidance is operationally specific — "PNG with transparency preferred so the stamp lands correctly on signed PDFs" tells admin why, not just what.
  • "Where this appears" gold-tinted usage list tells admin every surface the asset will show on. Logo: receipts, parent app, announcement headers, favicon. Stamp: GES return, annual census, transcripts, official letters. Admin understands the consequences of getting this wrong before uploading.
  • Stamp card has "Upload stamp" CTA in gold-primary while logo card has "Replace / Remove" because the logo is already uploaded. Different states get different action affordances; the missing-asset case is more prominent than the asset-management case.
  • Region 3 — Academic structure as a 5-row card with 200px label column + value column. Each row has a Fraunces label + italic gray sub-line + JetBrains Mono / Fraunces value. Same row pattern as the GES return identification grid — visual consistency.
  • Term bar row shows the 3 terms as horizontal pills with current Term 1 in gold-tinted bg. The "TERM IN PROGRESS" tag floating above the current term gives temporal context. The bars echo the calendar visual from attendance settings — same vocabulary.
  • "Click to view term-by-term calendar" sub-line on the current academic year row hints that drilling deeper opens the attendance calendar. Cross-linking between settings sections — the term calendar is configured in attendance settings but visible here as context.
  • "BECE 1—9 + percentage · 1 highest · 6 pass · 9 lowest" assessment scale row. Same scale-reminder pattern as the GES return and school stats. Settings page tells admin what convention is locked in for this school. Schools using BECE can't easily switch to a different scale mid-year; this row makes the lock visible.
  • Save bar at the bottom shows "No unsaved changes · Last edited 14 days ago by Florence" rather than the change-summary that appears when admin has draft edits. The bar is bi-state — clean state vs unsaved-changes state. Same save-bar pattern as attendance settings.
  • "View change log" CTA in the save bar reveals an audit history of who changed what and when. Settings changes are auditable — important for accountability when a setting affects compliance (statutory PDFs, payment routing).
  • "Edit fields" primary CTA flips the page into edit mode. Default state is read-only display; clicking Edit converts the field cards into editable inputs with Save / Discard. Read-first, edit-on-demand protects admin from accidental keypresses on consequential settings.
  • What's not on this page: No school motto, no mission statement, no "About us" copy. Operational settings only; marketing copy lives elsewhere (probably the website outside Omnischools). No file size limits in the format guidance because they're system-handled; admin doesn't need to think about them.
  • What's also not here: No "preview my stamp on a sample receipt" interaction. The stamp preview shows the asset; admin trusts that the system will apply it correctly to PDFs. Adding a preview-in-context flow would be useful but adds complexity for marginal gain.
